# Приватность и защита данных

Приватность — надмножество безопасности данных: даже хорошо защищённые данные становятся проблемой приватности, если их не нужно было собирать с самого начала. Пять вопросов к каждому элементу персональных данных:

1. Нам это нужно? (Нет — удалите поле.)
2. Нам это нужно навсегда? (Нет — политика retention.)
3. Кто должен иметь возможность читать? (Минимум — узкий IAM, отдельные ключи.)
4. Как доказать (аудитору и пользователю), что мы с этим делаем? (Audit trail.)
5. Как удалить по запросу? (Пользовательские права — access, export, delete.)

## Минимизация

- Собирайте только поля, которые требует фича. Email, который используется только для сброса пароля, не требует номера телефона.
- Избегайте «на всякий случай» логирования тел запросов, пользовательского ввода и содержимого документов.
- Псевдонимизируйте, где возможно — непрозрачный user ID обычно достаточен для аналитики; email — нет.

## Криптография

- TLS с HSTS везде. См. `rule-additional-cryptography.md`.
- Шифрование at rest для БД, файловых хранилищ, бэкапов.
- Стойкое хэширование паролей с per-user солью (Argon2id / scrypt / bcrypt) — см. `rule-authentication-mfa.md`.
- Pinning сертификатов — только там, где подходит (мобильные приложения с контролируемым каналом обновлений) и с планом ротации. См. `rule-additional-cryptography.md`.

## Приватность транспорта

- Предпочитайте соединения, не утекающие метаданные: избегайте DNS-over-HTTP без TLS; используйте DoH или DoT, где доступно.
- Рассмотрите OCSP stapling, чтобы клиенты не обращались к CA за отзывом напрямую — сам запрос — телеметрия третьей стороне.
- Блокируйте загрузку стороннего контента по не-TLS-путям. Не-TLS-картинка на TLS-странице может утечь cookie через URL-параметры или вскрыть IP.

## Сессии и аутентификация — угол приватности

- Ответ на неверный логин — общий («Invalid username or password»), чтобы исключить перечисление аккаунтов.
- Flow сброса возвращает одинаковый ответ для существующих и несуществующих аккаунтов.
- Session ID генерируются через CSPRNG и хранятся на сервере; на клиенте — только session-cookie. См. `rule-sessions-cookies.md`.
- Не выдавайте user-идентификаторы в URL, `Referer`-заголовках или analytics-событиях. Пользуйтесь opaque-редиректами, `Referrer-Policy: no-referrer` и first-party-аналитикой, видящей только нужные поля.

## Прозрачность

- Опубликуйте privacy policy: какие данные собираются, зачем, сколько хранятся, с кем шарятся.
- В продукте — just-in-time-подсказки в точке сбора, не только зарытые в ToS.
- Для сторонних интеграций (аналитика, A/B, реклама) — перечислите по именам и целям. Предлагайте opt-out, где требует юрисдикция.
- Информируйте о лимитах — «Этой фиче нужен ваш город для провайдера погоды, не точная локация».

## Пользовательские права

Поддерживайте как минимум:

- **Access** — пользователь видит, какие данные вы о нём храните.
- **Export** — машиночитаемая выгрузка.
- **Delete** — пользователь может стереть. Часто корректная реализация — «псевдонимизация»: цепочка событий, ссылающихся на user ID, tombstone-ится, а строка в users чистится.
- **Correct** — пользователь может исправить неверные данные.

Автоматизируйте эти flow, где объём того требует. Вручную обрабатываемые data-rights-запросы становятся непоследовательными и медленными.

## Audit trail для приватности

- Логируйте обращения к чувствительным записям (кто смотрел SSN пользователя, кто запускал bulk-экспорт).
- Держите лог отдельно от данных, которые он аудитирует — same-system-логи — мягкая точка для инсайдерской угрозы.
- Периодически ревизируйте; всплывайте аномалии (инженер внезапно читает каждую пользовательскую запись) в мониторинге.
- Когда data-rights-запрос выполнен, тоже логируйте — по ручке (handle), не сами данные.

## Шаринг данных со сторонами

- Контрактно: соглашения с вендором, DPA там, где требует юрисдикция.
- Технически: вендору — минимум полей. Analytics-вендору не нужен email; email-вендору не нужна история покупок.
- Ведите инвентаризацию шаринга — единый список «кто что получает» — это первый документ, который спросит аудитор.

## Чек-лист реализации

- У каждого собираемого поля в коде или конфиге задокументирована цель.
- Политика retention принудительно выполняется job-ом, а не надеждой. Lifecycle-правила на object-storage, TTL на DB-записях, плановые удаления.
- Шифрование in transit и at rest для всего выше класса «public».
- Обобщённые ошибки для аутентификации / enumeration-чувствительных flow.
- Сессии через CSPRNG, хранятся на сервере, cookie-only на транспорте.
- Audit trail для доступа к restricted-данным.
- Data-rights-flow реализованы end-to-end и прогоняются в тестах.
- Vendor-инвентарь свежий; DPA на руках.
