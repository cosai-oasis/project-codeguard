# Аутентификация и MFA

## Цели проектирования

Хорошая система аутентификации отражает атаки на учётные данные, держит секреты вне зоны досягаемости атакующего, предлагает сильные вторые факторы и даёт пользователю адекватный путь восстановления. Каждое решение ниже работает на одну из этих четырёх целей.

## Идентификаторы и UX

- **Внутренний идентификатор** — случайное, не-последовательное, не-публичное значение (UUID или эквивалент). Email/логин оставляйте только для UX; внутренний ID никогда не публикуйте в URL и токенах, уходящих клиенту.
- **Логин** — где возможно, разрешайте email *или* имя пользователя; email требует подтверждения владения перед тем, как стать путём входа.
- **Сообщения об ошибке** при неудачном входе — общие: «Неверный логин или пароль». Никогда не указывайте, что именно не совпало. Перечисление аккаунтов через регистрацию или сброс пароля — такая же плохая вещь, см. раздел «Восстановление».
- **Время отклика** должно быть примерно одинаковым для случаев (пользователь есть / его нет / пароль неверный). 50 мс разницы хватит, чтобы атакующий собрал каталог пользователей.
- **Дружелюбие к менеджерам паролей** — `<input type="password" autocomplete="current-password">`, разрешайте вставку, не блокируйте правый клик, не дробите поле на N односимвольных инпутов.

## Парольная политика

- Принимайте парольные фразы. Допускайте полный диапазон Unicode и любой печатаемый ASCII, включая пробелы.
- Минимальная длина — 8 символов; практический порог для новых систем — 12.
- Максимальная — не меньше 64; 128 — разумная верхняя граница для защиты от DoS. Никогда не обрезайте молча.
- **Не** вводите правила состава (одна заглавная, одна цифра, один спецсимвол). На практике они снижают энтропию, загоняя людей в предсказуемые шаблоны.
- Проверяйте новые пароли по брейч-корпусу — через k-anonymity API (HIBP Pwned Passwords), чтобы открытый текст не покидал сервер. Всё из списка — отклонять.

## Хранение паролей — хэшировать, а не шифровать

Выберите алгоритм, настройте параметры так, чтобы verify занимал ~500 мс – 1 с на вашем продакшен-железе, и подкручивайте их по мере роста вычислительной мощности.

| Алгоритм | Параметры (базовые) | Примечания |
|----------|---------------------|------------|
| **Argon2id** (предпочтительно) | `m=19–46 МиБ`, `t=2–1`, `p=1` | Современный, memory-hard, победитель PHC |
| **scrypt** | `N=2^17`, `r=8`, `p=1` | Старше, но всё ещё годится |
| **bcrypt** | `cost ≥ 10` | Легаси; помните про обрезку ввода на 72 байта |
| **PBKDF2-HMAC-SHA-256** | ≥ 600 000 итераций | Требуется только при FIPS-ограничениях |

Универсальные правила:

- Генерируйте уникальную соль на пользователя (минимум 16 байт из CSPRNG). Обычно это делает библиотека; проверяйте, что делает.
- Храните только `алгоритм + параметры + соль + хэш`, конкатенированные в каноническом формате библиотеки.
- Сравнивайте хэши константно-временной функцией. Штатные `verify` в крипто-библиотеках обычно корректны; ручное сравнение — обычно нет.
- При смене алгоритма или cost — *прозрачно перехэшируйте* на следующем успешном входе. Не принуждайте всех пользователей к сбросу.

Опциональный pepper (секрет, общий для приложения, вне БД — обычно в KMS/HSM):

- Применяйте `HMAC(pepper, password)` **перед** алгоритмом хэширования пароля или шифруйте получившийся хэш.
- Требует плана ротации/перехэша на случай утечки; без такого плана не добавляйте.

Unicode и null-байты должны пройти через ваш хешер неизменными. Некоторые реализации bcrypt молча обрываются на первом NUL — проверяйте.

## Усиление flow логина

- HTTPS на каждом auth-эндпоинте; HSTS — сайтово.
- Rate-limit: по IP, по аккаунту и глобально. Credential stuffing идёт одновременно с тысяч IP — только по-IP лимиты не спасут.
- Предпочитайте throttling с прогрессивным backoff-ом жёстким блокировкам. Перманентные lockouts — DoS-оружие против собственных пользователей.
- Альтернативная антиавтоматика (CAPTCHA, proof-of-work) допустима как эскалация, а не как дефолт.
- Коды статусов, тела ответов, тайминги и downstream-пути должны быть одинаковыми во всех случаях неудачи.

## Многофакторная аутентификация

### Иерархия факторов

| Уровень | Факторы | Куда |
|---------|---------|------|
| Сильные (устойчивые к фишингу) | WebAuthn / passkey, FIDO2-ключи | Дефолт для чувствительных аккаунтов |
| Приемлемые | TOTP (RFC 6238), смарт-карта + PIN | Базовый |
| Слабые — избегать при возможности | SMS, голосовой звонок, email-код | Только когда ничего лучше нет |
| Не использовать | Контрольные вопросы | Считать непрозрачными данными, а не фактором аутентификации |

### Когда требовать MFA

Минимум: вход, смена пароля или основного email, отключение MFA, эскалация привилегий, high-value транзакции, любой вход с нового устройства/локации/диапазона IP. Адаптивные сигналы (fingerprint устройства, гео-скорость, репутация IP, совпадение со списком сломанных паролей) должны снижать порог ещё.

### Восстановление

- Генерируйте одноразовые backup-коды при enrollment-е. Поощряйте регистрацию более одного фактора.
- Сброс MFA должен требовать сильной идентификации — тот же путь, которым атакующий будет захватывать аккаунт.
- После неудачной MFA-попытки предлагайте альтернативные зарегистрированные методы, логируйте попытку с контекстом (user/IP/устройство) и оповещайте пользователя.

## Федерация — OAuth 2.0 / OIDC / SAML

Не изобретайте свой федеративный протокол. Берите библиотеку, реализующую спецификацию, потом прочитайте саму спецификацию — чтобы понимать, что библиотека защищает, а что нет.

### Чек-лист OAuth 2.0 / OIDC

- Для публичных клиентов (SPA, мобилка, desktop) — **Authorization Code + PKCE**. Не используйте Implicit или Resource-Owner-Password-Credentials.
- Валидируйте `state` на каждом колбеке; `nonce` — на ID-токенах.
- `redirect_uri` — строгое совпадение: никакой substring, префикса или wildcard. Защищайте обработчик колбека от open-redirect.
- Привязывайте токены к `audience` и набору `scopes`; и то и другое проверяйте на принимающей стороне.
- Где экосистема позволяет — sender-constrained токены (DPoP или mTLS).
- Refresh-токены ротируйте при использовании; отзывайте при logout, смене пароля или рисковом сигнале.

### Чек-лист SAML

- TLS 1.2+; подписывайте ответы и ассершны; шифруйте assertion-ы с чувствительным содержимым.
- Валидируйте `Issuer`, `InResponseTo`, `NotBefore` / `NotOnOrAfter`, `Recipient`, `Destination` по ожидаемым значениям.
- Проверяйте подписи по запиненному набору доверенных ключей — а не по тому, что говорит ответ.
- Защищайтесь от XML Signature Wrapping: строгая схема-валидация, извлекайте assertion-ы по ссылке на подписанный элемент, а не XPath-ом по всему документу.
- Предпочитайте SP-initiated flow; валидируйте `RelayState`; детектируйте replay.

## Токены — JWT vs opaque

Предпочитайте **opaque-токены** и серверное хранилище везде, где важна возможность отзыва. «Stateless» не стоит операционной стоимости украденного токена, который нельзя аннулировать.

Если уж JWT:

- `alg` фиксируйте на сервере — не доверяйте заголовку. `none` отклоняйте.
- Валидируйте `iss`, `aud`, `exp`, `iat`, `nbf` на каждой проверке.
- Короткие сроки жизни (5–15 минут для access-токена). Refresh с ротацией.
- Подписывайте ключом из KMS/HSM; для HMAC-токенов — сильный секрет вне бинарника.
- Подумайте о привязке токена к контексту клиента (например, хэш device-идентификатора в HttpOnly cookie), чтобы украденный токен не работал на чужом клиенте.
- Реализуйте deny-list на logout и критические события, даже если это подрывает «stateless» — это то, что хранит пользователей.

## Сброс пароля / восстановление аккаунта

- Ответ возвращайте одинаковый, существует аккаунт или нет. Идите одной кодовой веткой.
- Токен сброса — криптосекрет: ≥ 32 байт из CSPRNG, хранится как хэш (не открытый), одноразовый, короткий TTL (15–60 минут).
- Ссылка сброса ведёт на HTTPS-URL на закреплённом доверенном домене. На UI сброса — `Referrer-Policy: no-referrer`, чтобы токен не утёк в `Referer`, если пользователь кликнет по ссылке со страницы сброса.
- После успешного сброса:
  - инвалидируйте все существующие сессии пользователя,
  - требуйте повторную аутентификацию,
  - оповещайте пользователя на всех доступных каналах,
  - не логиньте автоматом.
- Никогда не блокируйте аккаунт за сам *запрос* сброса — это вектор DoS. Лимитируйте и алёртьте.

## Админские / внутренние аккаунты

- Админский вход — на отдельном пути (отдельный поддомен, отдельный сервис).
- Сильные факторы обязательны (WebAuthn / железный ключ, не TOTP).
- Проверки postur-а устройства, IP-allow-list, step-up перед чувствительными операциями.
- Отдельные сессионные cookie и короче таймауты.

## Сигналы мониторинга

Логируйте (структурно, без PII где возможно):

- успех/неудача логина, с reason-кодом,
- enrollment MFA и каждую verify-попытку,
- смены пароля / email,
- блокировки и backoff-ы,
- выпуск/отзыв токенов,
- оповещения «новое устройство» / «новое гео».

Никогда не логируйте: сырые пароли, сырые MFA-коды, сырые session-ID, сырые токены, ответы на секретные вопросы.

Детектируйте: credential stuffing (много аккаунтов, мало паролей на аккаунт, много IP), impossible travel, brute-force по токенам сброса.

## Чек-лист реализации

- Хранение паролей: Argon2id предпочтительнее, per-user соль, constant-time verify. Breach-check на set/change.
- MFA: WebAuthn или железные ключи для высокого риска; TOTP как базовый; SMS — только когда больше ничего.
- Федерация: Authorization Code + PKCE, строгое совпадение redirect_uri, audience/scope проверяются, refresh-ротация.
- Токены: короткоживущие, sender-constrained где возможно, отзыв работает, ключи в KMS/HSM.
- Сброс: одноразовый хэшированный токен, одинаковые ответы, re-auth при сбросе, сессии ротируются.
- Защита от абьюза: rate-limit на всех auth-эндпоинтах, одинаковые ошибки, мониторинг аномалий.

## План тестирования

- Login: happy path, неверный пароль, несуществующий пользователь, заблокированный пользователь — все должны возвращать один и тот же 401 и схожий тайминг.
- MFA: enroll, verify, recover, fallback — каждый с негативами.
- OAuth: PKCE обязателен, `state` валидирован, `redirect_uri` strict-match, подпись `id_token` и `nonce` проверяются.
- Токен: replay после logout должен падать; несовпадение audience должно падать; `alg: none` должен падать.
- Сброс: одинаковый ответ для существующего и несуществующего пользователя; токен одноразовый; ротация сессий после сброса.
