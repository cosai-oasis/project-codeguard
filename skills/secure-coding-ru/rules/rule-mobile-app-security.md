# Безопасность мобильных приложений

Мобильное устройство — враждебная среда. Пользователь может быть противником (исследователи, конкуренты, любопытные владельцы), ОС может быть скомпрометирована (rooted / jailbroken), а сеть между — часто недоверенная.

## Архитектурные принципы

- **Security-решения — на сервере.** Никогда не доверяйте клиенту ни в аутентификации, ни в авторизации, ни в ценах, ни в остатках, ни в чём-либо, что бизнесу важно. Клиент показывает; сервер решает.
- **Наименьшие привилегии** на каждом слое — запрашиваемые разрешения, объявленные capabilities, выданные entitlements.
- **Эшелонированная защита** — считайте, что каждый слой будет пробит.
- **Только доверенные библиотеки** — у мобильных SDK долгая история неприятных сюрпризов. Проверяйте до интеграции; мониторьте после.
- **Безопасный путь обновлений** — приложение должно быть обновляемо, и должен быть механизм forced-update под security-критичные патчи.

## Аутентификация и авторизация

- Flow аутентификации — стандартные: OAuth 2.0 с PKCE, OIDC. См. `rule-authentication-mfa.md`.
- Пароли на устройстве не хранить. Используйте отзываемые access-токены с коротким временем жизни и ротацией refresh.
- Креды никогда не хардкодятся в бинарник (см. `always-no-hardcoded-secrets.md`). Transport-layer identity приложения — исключение: узко scoped сервисный API-key, завёрнутый в secure-storage платформы, допустим; широкий кред — нет.
- Каждый запрос к бэку аутентифицирован; авторизация проверяется на сервере на каждом запросе.
- Session-таймаут + remote logout. Step-up-аутентификация для чувствительных операций.
- **Биометрия** — через платформенный API (Face ID / Touch ID на iOS, `BiometricPrompt` на Android). Фоллбэк — device passcode, а не самодельный PIN. Биометрия защищает серверный токен; это не аутентификация к серверу.
- Избегайте 4-значных PIN-ов; где PIN разрешён — требуйте сложность passphrase.

## Хранение данных на устройстве

- Храните только то, что нужно приложению. Остальное — на сервере.
- Чувствительные данные — в платформенном secure-хранилище:
  - iOS: **Keychain** (с нужным accessibility-классом — обычно подходит `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`) и **Secure Enclave** для ключевых операций.
  - Android: **Keystore**, в идеале hardware-backed (TEE или StrongBox). Для data-at-rest помимо ключей — `EncryptedSharedPreferences` / `EncryptedFile`.
- Никогда не храните креды или токены в:
  - `UserDefaults` / `NSUserDefaults` (iOS) — не зашифровано.
  - `SharedPreferences` (Android) — world-readable на root-устройствах.
  - Лог-файлах приложения.
  - Незашифрованных SQLite-БД.
- Приватным файлам — только internal storage; external-storage на многих версиях Android читается всеми.
- Отключайте автоматический backup для файлов с секретами — `android:allowBackup="false"` с явным backup-rules-файлом или iOS-класс защиты данных `kCFURLIsExcludedFromBackupKey`.
- Отключайте кэш снапшотов, когда видны чувствительные экраны (iOS: blur/blank на `applicationWillResignActive`; Android: `FLAG_SECURE` на чувствительных Activity).
- Политики кэша: не кэшируйте тела ответов с кредами. `Cache-Control: no-store` со стороны сервера плюс тюнинг `URLCache` / OkHttp-кэша на клиенте.

## Минимизация PII

- Собирайте только поля, нужные для фичи.
- Чистите по расписанию — сообщения чата истекают, история поиска ротируется.
- Информируйте пользователя и отрабатывайте export / delete-запросы.

## Сеть

Считайте сеть враждебной.

- Только HTTPS, с валидацией сертификата (никакого `allowsInsecureHTTPLoads`, никакого `trust-all` custom-trust-manager-а).
- Сильные cipher suites; TLS минимум 1.2, предпочтительно 1.3.
- Сертификат — от CA, которому доверяет ОС; для корпоративных сертификатов настраивайте доверие явно через Network Security Config (Android) или App Transport Security (iOS).
- Для ценных приложений рассмотрите **pinning сертификатов** — см. `rule-additional-cryptography.md`. Канал ротации планируйте до развёртывания.
- Дополнительное шифрование чувствительных payload-ов поверх TLS, когда данные очень чувствительны — скомпрометированный CA не должен прочитать всё.
- Не полагайтесь на SMS для security-критичных доставок. SMS тривиально перехватывается в современных сетях.

## Целостность кода и anti-tamper

- **Release-сборки**:
  - Debuggable off. Никакого verbose-логирования пользовательских данных.
  - Обфускация (ProGuard / R8 на Android, LLVM-bitcode-обфускация плюс меры в стиле SwiftShield на iOS). Обфускация — задержка, а не защита, но задержка имеет значение.
  - Стрипните символы.
- **Runtime-целостность** (там, где требует модель угроз):
  - Детекция отладчика.
  - Детекция code-injection / method-hooking (индикаторы Frida / Cydia Substrate).
  - Детекция эмуляторов и rooted / jailbroken устройств. Примите, что это игра в кошки-мышки; деградируйте мягко, не крешитесь.
  - Проверяйте подпись приложения в рантайме.
- Используйте платформенный API integrity-attestation:
  - Android: **Play Integrity API**.
  - iOS: **App Attest** и **DeviceCheck**.

Они выдают подписанные токены, которые сервер может верифицировать перед высокорисковой операцией.

## Платформенные нюансы

### Android

- `EncryptedSharedPreferences` / `EncryptedFile` (AndroidX Security). Ключи — в `AndroidKeyStore`, в идеале `StrongBox`.
- `android:exported="false"` на activity, service, provider, receiver по умолчанию. Экспортируйте только то, что обязано, и защищайте permissions или signature-check-ами.
- Интенты с чувствительными данными — explicit component name, не implicit intent.
- Отключайте backup для файлов с секретами (`android:allowBackup="false"` + `fullBackupContent`).
- Network Security Config явно перечисляет домены и trust-anchor-ы, которые использует приложение.

### iOS

- Чувствительные предметы — в Keychain со строжайшим работающим accessibility-классом.
- Secure Enclave — для асимметричных ключей device-binding-а.
- Отключайте Siri-intent-ы, выставляющие чувствительное, либо ставьте им `requiresUserAuthentication = true`.
- Виджеты и Live Activities, показывающие чувствительное, маскируют его на lock-screen.
- Deep-link-и валидируются — вредоносный URL-scheme-вызов не должен авто-навигировать в чувствительные flow.
- Не храните чувствительное в `Info.plist` или другом plist.

## Ввод / вывод на UI

- Валидация и санитизация и на клиенте (UX), и на сервере (безопасность).
- Маскируйте чувствительные поля (пароль, PIN, OTP), чтобы защищать от shoulder-surf. Отключайте clipboard на по-настоящему чувствительных полях.
- Security-события, видимые пользователю, — «вход с нового устройства», «смена пароля», «смена email» — приходят через независимый канал.

## Тестирование и мониторинг

- Пентест с crypto-review и jailbroken / rooted устройством.
- Автоматические security-тесты в CI — проверить, что pinning включён, что integrity-checks на месте, что приложение отказывается запускаться на debuggable-сборках в продакшене.
- Real-time мониторинг бэкенд-сигналов, указывающих на компрометацию клиента (необычные session-паттерны, провалы integrity-attestation, CI-подписанные vs атакующим подписанные бинарники).
- План инцидент-реакции покрывает потерянные устройства, компрометацию кредов и импостеров в магазинах.

## Кратко

Большая часть мобильной безопасности — аккуратное использование платформы: Keychain/Keystore, App Transport Security, integrity-API, hardware-backed-ключи. Большинство провалов в поле — от срезанных углов: токен в `UserDefaults`, кастомный TLS-trust-manager, принимающий любой сертификат, root-detection, отключённый для отладки и не включённый обратно. Составьте список таких shortcut-ов на ревью и не деплойте их.
