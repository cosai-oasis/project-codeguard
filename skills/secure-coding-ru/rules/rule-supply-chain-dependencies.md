# Безопасность зависимостей и цепочки поставок

Любой сторонний пакет работает с привилегиями импортирующего процесса. Минорный бамп с вредоносным кодом, typosquat, скомпрометированный аккаунт мейнтейнера — сейчас это рутина, а не редкость. Контроли разбиваются на четыре ведра: **знать, что тянете; тянуть только из доверенных источников; детектить изменения; иметь план, когда что-то пошло не так.**

## Политика и governance

- Allow-list на registry и scope. `@my-company/*` только из внутреннего registry; публичный — только как именованный фолбэк для явно объявленных публичных пакетов.
- Требуйте lockfile. Версии пинятся точно. Для контейнерных образов — пин по **digest** (`image@sha256:...`), не по тегу: теги изменяемы.
- Генерируйте **SBOM** для каждого приложения и каждого образа. Храните с артефактом. (SPDX или CycloneDX.)
- Аттестуйте **provenance** сборки — SLSA уровень ≥ 2 выпускает подписанное утверждение о том, что и из какого источника собрано.
- Документируйте, какие версии зависимостей приемлемы для какого типа изменений (major-бампы через ревью, патчи автоматом).

## Гигиена пакетов

### Применимо в любой экосистеме; npm — как конкретный пример

- Гоняйте SCA регулярно (`npm audit`, Dependabot, Renovate, Snyk, GitHub Advanced Security). SLA по серьёзности: критика — часами, high — неделями.
- **Детерминированный install в CI**: `npm ci`, не `npm install`. `yarn install --frozen-lockfile`. `pip install --require-hashes`. `cargo install --locked`.
- Install-скрипты — вектор атаки. Там, где экосистема позволяет, отключайте (`npm config set ignore-scripts true`) и включайте обратно только для нескольких пакетов, которым это реально нужно.
- Приватные registry задаются scope-ом в `.npmrc` / `pip.conf` / `.cargo/config.toml`. Wildcard-фолбэк на публичный registry для внутренних scope — отключён.
- Включайте verification целостности. `npm ci` делает это через поле `integrity` в lockfile.
- **Publish-гигиена** — 2FA на каждом maintainer-аккаунте, имеющем право публиковать пакеты вашей организации. Отзывайте неиспользуемые токены.

## Практики разработки

- Минимизируйте зависимости. Добавить библиотеку — добавить атакующего. Для тривиальных утилит обычно лучшая стратегия — стандартная библиотека или первопартийный код.
- Предпочитайте пакеты с активным сопровождением и значимым количеством загрузок. Свежий single-maintainer-пакет с единственной фичей должен проходить планку повыше, прежде чем его принять.
- Следите за **typosquat** и **dependency confusion**:
  - `requests` vs `request` vs `reuqests` — опечатка, вредоносно.
  - Публичный пакет с именем вашего внутреннего scope обходит плохо настроенный resolver.
- Мониторьте **protestware** — пакеты, в чей более поздний релиз мейнтейнер вставляет разрушающее поведение. Пиньте и ревизируйте.
- Герметичные сборки — никакой сети во время компиляции и упаковки, кроме известных кэшей. Всё, что приходит через кэш, проверяется по целостности.

## Интеграция с CI/CD

См. `rule-ci-cd-containers.md`. Релевантно этому правилу:

- SCA, SAST и IaC-сканеры работают как гейт. Критики блокируют merge.
- Артефакты подписаны; подписи проверяются на деплое. На Kubernetes политики принуждает admission-контроллер.
- PR с обновлениями зависимостей тестируются как любые другие — зелёный lockfile — не повод мёржить.

## Управление уязвимостями

### Пропатченное CVE

- Тестируете и деплоите апдейт. Документируете API-ломающие изменения, чтобы потребители были в курсе.
- Для пропатченного transitive — лучше обновлять top-level-зависимость; pinned override на transitive — издержки на поддержку.

### Непропатченное CVE

- Валидация ввода, безопасные обёртки, feature-flag для отключения уязвимого пути.
- Virtual-patching на WAF (`rule-ci-cd-containers.md`). Правило WAF снимаем, когда код пропатчен.
- Мониторим upstream на фикс; ведём issue.

### Принятие риска

- Документируйте, почему принятие — правильный выбор, с бизнес-обоснованием и компенсирующим контролем.
- Эскалируйте на соответствующую роль (security, engineering leadership). Недокументированное принятие — это неисправленная уязвимость.

## Инцидент-реакция

- Быстрый rollback по каждому артефакту — старый образ доступен, старый бинарник доступен.
- Возможность изолировать скомпрометированные пакеты — deny-list в registry, чтобы ни одна сборка их больше не тянула.
- Throttled-rollout для новых версий, когда уверенность низка.
- Плейбук для оповещения стейкхолдеров — кто говорит клиентам, каким каналом, с каким SLA.

## Threat intelligence

- Подпишитесь на advisory-фиды экосистем, которые используете (`npm security`, `pypi announce`, `rustsec`, distribution advisories).
- Автоматически заводите тикеты на критические CVE, задевающие зависимости из вашей инвентаризации.
- Держите SBOM запрашиваемым — когда новое CVE приходит, вопрос «используем ли мы это?» должен быть grep-ом, а не аудитом.

## Чек-лист реализации

- Lockfile присутствует и уважается в CI (`npm ci` и эквиваленты в других местах).
- Integrity-checks / hash-verification включены.
- Приватные registry scoped; публичный не является wildcard-фолбэком.
- SBOM генерируется и хранится с артефактом; provenance аттестуется, где инструменты позволяют.
- Подписи проверяются до деплоя; admission-контроллеры enforce-ят.
- Автоматические обновления зависимостей с тестами и ревью-гейтами.
- SLA на remediation по серьёзности; просроченные — эскалируются.
- Rollback и процедуры изоляции протестированы, а не только написаны.
