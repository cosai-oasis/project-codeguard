# Авторизация и контроль доступа

## Ментальная модель

Аутентификация отвечает на вопрос *«кто это?»*. Авторизация — на вопрос *«разрешено ли именно это действие над именно этим объектом прямо сейчас?»*. Эти две вещи должны быть раздельными — и авторизация должна проверяться на *каждом* запросе, а не только при логине или на шлюзе.

## Четыре принципа, по приоритету

1. **Default deny.** Если ни одно правило не разрешает действие явно — ответ `403 Forbidden`. Добавление правила — это всегда сознательное решение.
2. **Наименьшие привилегии.** Выдавайте минимум необходимых прав. Разрешения расползаются со временем молча; закладывайте регулярный pruning, а не разовую выдачу.
3. **Проверяйте на каждом запросе.** Не предполагайте, что дошедший до эндпоинта вызов авторизован там бывать. Middleware, route-filter или явные проверки в начале каждого хендлера — что-то одно, последовательно.
4. **ABAC / ReBAC вместо RBAC, когда модель ресурсов имеет связи.** Роли разваливаются, как только встаёт «этот пользователь может редактировать документ, потому что он коллаборатор родительского проекта». Роль этого не выражает — атрибут или связь выражают.

## Системные шаблоны

- Централизуйте точку принятия решения. Разбросанные по 40 хендлерам `if user.role == 'admin':` уходят в разнобой. Кладите политику в middleware, filter или PDP (policy decision point), с которым сверяется каждый хендлер.
- Ограничивайте запросы данных скоупом субъекта. `currentUser.documents.find(id)` безопаснее, чем `Document.find(id)`, потому что границу обеспечивает БД.
- На «не авторизован» и «не существует» возвращайте одинаковый ответ (часто `404` для GET, `403` для небезопасных методов), чтобы API не перечислял ID ресурсов.
- Каждый отказ логируйте с `user_id`, действием, типом и ID ресурса (или стабильным хэшем) и коротким rationale-кодом. Дашборды на этих данных показывают и атаки, и баги.

## Профилактика IDOR (Insecure Direct Object Reference)

IDOR возникает, когда эндпоинт принимает ID от вызывающего и смотрит объект, *не* сверив отношение к нему.

Сломано:

```python
# POST /api/orders/{order_id}/cancel
order = Order.objects.get(pk=order_id)  # отменит чей угодно заказ
order.cancel()
```

Исправлено:

```python
order = Order.objects.get(pk=order_id, customer=request.user)  # иначе DoesNotExist
order.cancel()
```

### Указания

- Всегда разрешайте объект через запрос, ограниченный пользователем, или фетчите по ID и затем проверяйте владение *до* любого side-эффекта.
- Используйте неперечислимые идентификаторы (UUID v4, случайные токены) как defense in depth. Последовательные integer-ID — подарок атакующему.
- Проверки на уровне объекта должны быть в *сервисном* слое, не только на шлюзе — другие пути кода могут добраться до объекта мимо шлюза.

## Профилактика mass assignment

Не связывайте тело запроса напрямую с доменным объектом. Поля, которые должны контролироваться сервером (`role`, `is_admin`, `balance`, `created_at`, `user_id`), молча примут ввод атакующего.

Используйте DTO или явные списки полей:

Java (Jackson):

```java
public class UpdateProfileRequest {
    @JsonProperty("display_name") public String displayName;
    @JsonProperty("bio")          public String bio;
}
```

Python (Django serializer):

```python
class UpdateProfileSerializer(serializers.ModelSerializer):
    class Meta:
        model = UserProfile
        fields = ['display_name', 'bio']  # явный список — никогда "__all__"
```

Ruby on Rails:

```ruby
params.require(:profile).permit(:display_name, :bio)
```

Подходы на deny-list (`.except(:role)`) хрупкие — любое добавленное позже чувствительное поле оказывается открытым по умолчанию. Каждый раз — allow-list.

## Step-up / пер-транзакционная авторизация

Для операций с высоким импактом — перевод средств, экспорт всех данных, выдача ролей, отключение MFA — требуйте второй, специфичный для транзакции апрув:

- Показывайте пользователю именно те критические поля, которые он одобряет (сумма, получатель, область). Это «What You See Is What You Sign» — подтверждение должно отражать именно ту транзакцию, которая будет выполнена.
- Генерируйте уникальный, короткоживущий, одноразовый authorization-токен, привязанный к транзакции. Любое изменение данных транзакции должно инвалидировать токен.
- Step-up проверяйте на сервере, не в UI. Запрос без валидного step-up-токена отклоняется вне зависимости от того, что сделал UI.
- При провале перезапускайте весь flow — не давайте атакующему повторять только шаг одобрения.

## Тестирование и автоматизация

Держите матрицу авторизации как данные — YAML или JSON со списком эндпоинтов × роли × ожидаемый исход. Фрагмент:

```yaml
- path: /api/admin/users
  method: GET
  expected:
    admin: allow
    editor: deny
    viewer: deny
    unauthenticated: deny

- path: /api/orders/{id}
  method: DELETE
  expected:
    admin: allow
    owner_of_order: allow
    other_customer: deny
    unauthenticated: deny
```

Тестовый набор итерирует матрицу, выпускает токен для каждой роли, отправляет запрос и проверяет allow/deny. Негативные тесты (подмена ID, downgrade роли, отсутствие скоупа, replay токена после ротации) обязательны.

## Чек-лист реализации

- На каждом маршруте есть проверка авторизации — ни один маршрут не разрешён неявно.
- Проверка происходит в сервисном слое, не только на шлюзе.
- Запросы ограничены скоупом owner/tenant; прямые lookup-ы по ID всегда проверяют связь до любого side-эффекта.
- Биндинг запроса в модель — через DTO или явные allow-list-ы; чувствительные поля не биндятся.
- Step-up-авторизация прикручена для привилегированных операций с уникальными, короткоживущими, одноразовыми токенами.
- Матрица авторизации существует как данные и прогоняется в CI; сбои блокируют merge.
- Отказы логируются и показаны на дашбордах.
