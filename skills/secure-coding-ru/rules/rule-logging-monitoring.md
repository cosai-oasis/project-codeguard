# Логирование и мониторинг

Логи — основной след того, что делала система и что делал атакующий дальше. Они должны быть полезны для детекции и форензики, структурны для запросов и аккуратны, чтобы самим не стать дампом кредов.

## Что логировать

Минимум:

- События аутентификации — успех, неудача, reason-код (единообразный), MFA-события.
- Решения авторизации — denies целиком, allows на уровне ресурса для high-value-действий.
- Административные действия — назначения ролей, изменения прав, изменения конфигурации audit-log.
- Изменения конфигурации — приложения и инфраструктуры.
- Обращения к чувствительным данным — чтения и записи хотя бы для restricted-класса.
- Сбои валидации ввода — паттерн и payload (с замаскированными секретами).
- Ошибки безопасности — сбои криптоопераций, сбои валидации сертификатов, сбои verify-токенов.

Каждое событие несёт:

- Стабильный correlation / trace ID (пронизывающий сервисы).
- User ID (или стабильный псевдоним), хэш session ID — никогда сырого.
- Source IP и user agent.
- Идентификатор тенанта / организации.
- Метку времени UTC в RFC 3339 с миллисекундной точностью.
- Event type / name из маленького закрытого словаря.

## Как логировать

### Структура

Используйте JSON (или эквивалент структурного формата) с устойчивыми именами полей. Свободные строки — будущая проблема поиска, о которой пожалеете.

```json
{
  "ts": "2026-04-15T13:45:09.123Z",
  "level": "warn",
  "event": "auth.login.failed",
  "request_id": "req-8f12...",
  "user_id": "u-7c9a...",
  "source_ip": "203.0.113.42",
  "user_agent": "Mozilla/5.0 ...",
  "reason_code": "invalid_credentials"
}
```

### Log-injection

Любые данные, попадающие в запись лога из-за границы доверия, могут содержать переводы строк, разделители полей или управляющие символы, ломающие downstream-парсеры. Срезайте или экранируйте CR, LF и разделители формата. Структурные логгеры фреймворка делают это корректно — ручная конкатенация обычно нет.

### Редактирование

В логи никогда не попадают:

- Сырые пароли, хэши паролей, MFA-коды, recovery-коды
- Session ID (логируйте соль+хэш, если нужна корреляция)
- API-ключи, токены, OAuth-коды, JWT
- Полные номера карт, счетов, SSN — применяйте маскирующий фильтр
- Тела запросов эндпоинтов с кредами или health-data

Редактирование стройте как allow-list имён полей (что логировать) или deny-list (что выбрасывать), применяемый на сериализации — чтобы разработчик не обошёл его, написав в не тот логгер.

### Целостность

- Append-only хранилище или WORM, где требует комплаенс. Детекция подделки (hash-chain, почасовые Merkle-roots) для audit-логов.
- Централизованная агрегация — важные логи живут не на хосте, который их породил. Иначе атакующий, получивший root на хосте, может заметать следы.
- Контроль доступа к лог-стору — compute-аккаунт, *пишущий* логи, не может их *изменить*.

## Детекция и алёрты

Алёрт на:

- Аномалии аутентификации — credential stuffing (много пользователей × мало паролей × много IP), impossible travel, всплески неудачных логинов, повторные провалы MFA.
- Изменения привилегий — выдача ролей, новые admin-аккаунты, смена прав сервис-аккаунтов.
- Избыточные фэйлы валидации — признак прощупывания атакующим.
- Индикаторы SSRF — исходящие запросы в приватные диапазоны или неожиданные домены.
- Паттерны эксфильтрации — массовые чтения, необычные размеры ответов, новые bulk-egress направления.
- Сбои криптографии — повторные провалы проверки подписей, TLS-handshake-сбои с одного клиента.

У каждого алёрта — ранбук с плейбуком под сигнал. Алёрты, не приводящие к действиям, — шум, их перенастраиваем или удаляем.

## Хранение и защита

- Изолируйте хранилище логов от хранилища приложения. Отдельный раздел, отдельная БД, отдельный аккаунт в облаке.
- Файлы и каталоги с жёсткими правами; не под web-root.
- Синхронизированное время на всех системах (NTP) — форензика требует согласованных timestamp-ов.
- TLS на лог-пайплайне. Отправитель аутентифицируется end-to-end.

## Приватность и комплаенс

- Ведите data-inventory — какие персональные данные попадают в логи, где, зачем.
- Минимизируйте PII. Идентификатор пользователя, не являющийся email, обычно достаточен для диагностики.
- Retention — по политике (юридический, контрактный, операционный). Дефолт — «меньше полей, если они не нагружены».
- Поддерживайте user-linked удаление, где требует регуляция (GDPR right to erasure). Обычно это означает псевдонимизацию записей, а не их удаление; хэш-индекс уходит вместе с учёткой пользователя.

## Чек-лист реализации

- JSON-логирование везде; устойчивые имена полей; общий correlation ID.
- Митигации log-injection присутствуют в logging-пути.
- Редактирующие фильтры активны; покрытие проверено юнит-тестом, который пытается залогировать кред и проверяет, что он замаскирован.
- Централизованный лог-пайплайн, tamper-resistant-целевая точка, retention настроен.
- Security-алёрты определены, ранбуки связаны, периодически тестируются.
- Дашборды на ключевые сигналы (auth, privilege, data access).

## Валидация

- Юнит- и интеграционные тесты проверяют наличие ключевых полей и отсутствие полей-кредов.
- Периодические автоматические аудиты сканируют логи на паттерны секретов — те же regex, что алёртили бы на утечку в исходниках.
- Tabletop-упражнения, начинающиеся с лог-запроса и разворачивающие весь incident-workflow.
