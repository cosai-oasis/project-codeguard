# Гигиена сертификатов

## Правило

Всякий раз, когда агент сталкивается с данными сертификата — встроенными в исходник, загружаемыми из файла или возвращаемыми из вызова — он обязан: (1) осознать, что это сертификат, (2) проверить четыре ключевых свойства, перечисленных ниже, и (3) выдать находку с чёткой степенью критичности и конкретным руководством по устранению.

«Проверить» — значит: если содержимое доступно в контексте, распарсьте и отчитайтесь. Если есть только ссылка по пути к файлу или загрузка в рантайме — выдайте короткий ранбук, который ревьюер сможет выполнить, и предупреждение, если путь к коду валидацию обходит.

## Как распознать данные сертификата

### PEM-строки в исходнике

Ищите многострочные строковые литералы (константы, шаблоны), содержащие:

```
-----BEGIN CERTIFICATE-----
<base64-блоб>
-----END CERTIFICATE-----
```

Сопутствующие блоки, на которые тоже надо обращать внимание: `-----BEGIN RSA PRIVATE KEY-----`, `-----BEGIN EC PRIVATE KEY-----`, `-----BEGIN PRIVATE KEY-----` — это приватные ключи, они проходят по `always-no-hardcoded-secrets.md`.

### Чтение из файла

Типичные расширения: `.pem`, `.crt`, `.cer`, `.der`, `.p7b`, `.p12` / `.pfx`. Любое чтение подобных путей считайте событием «работа с сертификатом».

### Библиотечные вызовы

Любой из следующих сигналов означает, что в коде идёт обращение к сертификатам:

- OpenSSL: `PEM_read_X509`, `d2i_X509`, `SSL_CTX_use_certificate_file`
- Python: `cryptography.x509.load_pem_x509_certificate`, `ssl.SSLContext.load_cert_chain`
- Java: `CertificateFactory.getInstance("X.509")`, `KeyStore.load`
- Go: `tls.LoadX509KeyPair`, `x509.ParseCertificate`
- Node.js: `crypto.createCredentials`, `tls.createSecureContext` с параметрами `cert:` / `key:`
- .NET: `X509Certificate2`, `X509Store`

## Четыре проверки

Если содержимое сертификата доступно, дайте находки ниже. Если нет — порекомендуйте команду инспекции:

```
openssl x509 -in <путь_или_поток> -noout -text
```

(или платформенный эквивалент) и перечислите четыре проверки, которые выполнит ревьюер.

### Проверка 1 — Окно валидности

Возьмите `notBefore` и `notAfter`. Сравните с текущим временем.

- **Критично — просрочен**: `notAfter` в прошлом.
  > *«Сертификат истёк YYYY-MM-DD. TLS-клиенты будут отклонять соединение. Заменить до деплоя.»*
- **Предупреждение — ещё не валиден**: `notBefore` в будущем.
  > *«Валидность сертификата начинается YYYY-MM-DD. До этой даты он не будет принят.»*
- **Информация — скоро истечёт**: осталось меньше 30 дней.
  > *«Сертификат истекает YYYY-MM-DD (N дней). Планируйте ротацию.»*

### Проверка 2 — Стойкость открытого ключа

Проверьте алгоритм и размер модуля.

- **Высокий риск**: модуль RSA < 2048 бит.
- **Высокий риск**: ключ EC на кривой меньше P-256 (`secp192r1`, `P-192`, `P-224`).
- **Приемлемо**: RSA ≥ 2048 (для новых выпусков предпочтительно ≥ 3072), либо EC на P-256 / P-384 / Ed25519.

Отчёт:
> *«Открытый ключ сертификата — {алгоритм}-{размер}. Новые сертификаты должны использовать RSA ≥ 2048 или ECDSA на P-256 и выше. Перевыпустите перед деплоем.»*

### Проверка 3 — Алгоритм подписи

Смотрите поле `signatureAlgorithm`.

- **Высокий риск**, если алгоритм на базе MD5 или SHA-1 (`md5WithRSAEncryption`, `sha1WithRSAEncryption`).
- Приемлемо: семейства SHA-256 / SHA-384 / SHA-512.

Отчёт:
> *«Сертификат подписан {алгоритмом}, который не устойчив к коллизиям. Перевыпустите с подписью семейства SHA-2 (например, sha256WithRSAEncryption).»*

### Проверка 4 — Самоподписанность / издатель

Сравните `Issuer` и `Subject`.

- **Информация**: поля идентичны — сертификат самоподписан.

Самоподписанность не всегда небезопасна, но требует явной настройки доверия у каждого клиента. Отчёт:
> *«Это самоподписанный сертификат (Issuer = Subject). Допустим только там, где путь доверия управляется явно — внутренние сервисы, dev-окружения, приватные CA. Не использовать на публичных эндпоинтах.»*

## Дополнительная защита — флажите также

- **Приватный ключ рядом с сертификатом в репозитории.** Даже если сам сертификат в порядке, закоммиченный приватный ключ — критическая находка. Эскалируйте по `always-no-hardcoded-secrets.md`.
- **`verify=False`, `InsecureSkipVerify=true`, `rejectUnauthorized: false`, `NSURLSessionDelegate`, принимающий любой сертификат.** Любой код, отключающий верификацию, — критическая находка, потому что все четыре проверки становятся неактуальны в рантайме.
- **Pinning, принимающий любой backup.** Если pinning реализован, нужен хотя бы один резервный отпечаток; ноль резервов превратит ротацию в кирпичик для клиентов.

## Разобранные примеры

### Пример A — захардкоженный сертификат

```python
# Сертификат для legacy_service
LEGACY_CERT = """
-----BEGIN CERTIFICATE-----
MIIC...
-----END CERTIFICATE-----
"""
```

Отчёт:

> **Нужен security-ревью.** В `LEGACY_CERT` захардкожен сертификат.
>
> Выполните `openssl x509 -in - -noout -text` на декодированном содержимом и проверьте:
> - Валидность: `notAfter` не в прошлом, `notBefore` не в будущем
> - Стойкость ключа: RSA ≥ 2048 или ECDSA ≥ P-256
> - Подпись: семейство SHA-2 (не MD5 / SHA-1)
> - Issuer vs subject: задокументировать самоподписанность; принимать только для внутреннего / dev-использования
>
> **Отдельно:** захардкоженные сертификаты нельзя ротировать без изменения кода. Предпочтительно загружать из внешнего файла, примонтированного секрета или сертификатного стора.

### Пример B — загрузка с диска, верификация отключена

```go
cert, err := tls.LoadX509KeyPair("server.crt", "server.key")
// ...
client := &http.Client{Transport: &http.Transport{
    TLSClientConfig: &tls.Config{InsecureSkipVerify: true},
}}
```

Отчёт:

> **Критично.** `InsecureSkipVerify: true` полностью отключает проверку сертификата. Каждое соединение уязвимо к MITM. Удалите флаг; если вы обращаетесь к самоподписанному внутреннему сервису — добавьте соответствующий CA в пул RootCAs.
>
> Вторично: сертификат загружен из `server.crt`. Прогоните по этому файлу четыре проверки выше перед деплоем.

## При применении правила

В ответе либо:
- укажите, что данных сертификата в контексте не было, **либо**
- перечислите каждый сертификат / файл и исход каждой из четырёх проверок (или попросите ревьюера прогнать их).
