# Работа с файлами и загрузками

## Модель угроз

Эндпоинт загрузки принимает недоверенный бинарный контент из сети и пишет его в серверную ФС. Значение имеет каждая деталь — имя файла, расширение, заявленный content type, реальные байты, место хранения, URL, который в итоге его отдаёт, и кто имеет доступ к этому URL.

Защита *слоистая*: ни одна проверка сама по себе недостаточна. Вредоносный PNG может иметь валидный magic, нормальное расширение — и всё равно исполниться, если раздаётся из места, где он трактуется как код.

## Валидация расширений

- **Allow-list** на небольшой набор расширений, нужных фиче. Deny-list на `.exe` упускает `.pif`, `.bat`, `.sh`, `.jsp`, `.jspx`, `.phar`, `.phtml` и т. п.
- Валидируйте после декодирования имени — `.php%00.jpg` превращается в `.php\0.jpg` и может обрезаться до `.php` в зависимости от потребителя.
- Двойные расширения (`avatar.jpg.php`) отклоняйте, если приложение явно на них не рассчитано.
- Применяйте эту проверку после общей валидации ввода (длина, класс символов).

## Валидация Content-Type и magic-byte

- Заголовок `Content-Type` в загрузке — клиентский. Никогда не доверяйте ему как единственному источнику истины.
- Просматривайте реальные magic-bytes и сверяйте с allow-list-ом MIME. Библиотеки: `file-type` (Node), `python-magic`, Apache Tika, Go `net/http.DetectContentType` (как база — он нюхает только первые 512 байт, чего не всегда хватает).
- Принимайте загрузку, только если (заявленный MIME, «обнюханный» MIME, расширение) совпадают и все в allow-list.
- Sniff magic-byte — необходимо, но не достаточно: полиглот-файлы (валидный PNG + валидный JAR) его одного обходят.

## Обработка имени файла

- **Дефолт**: генерируйте новое имя файла (UUID v4 или похожее) на сервере и используйте везде. Оригинальное имя — метаданные, не компонент пути.
- Если оригинальное имя обязательно нужно сохранить:
  - Ставьте максимальную длину (255 байт для большинства ФС, 260 для путей Windows).
  - Ограничивайте набором `[A-Za-z0-9._-]` плюс пробелы. Всё остальное отклоняйте.
  - Запрещайте ведущий `.` (hidden-файлы, `.htaccess`) и ведущий `-` (ломает многие CLI-инструменты).
  - Сжимайте подряд идущие `.` (`screenshot...png` → `screenshot.png`) и явно блокируйте `..`.

## Валидация контента

- Для изображений — **перекодируйте** через библиотеку (`Pillow`, `libvips`, `ImageMagick` с ограниченными policies). Пережатие сохраняет картинку и выкидывает всё паразитное, спрятанное в метаданных или между чанками. Это, с отрывом, самая эффективная защита для аплоадов изображений.
- Для Microsoft Office / PDF — валидируйте вендорским проверенным парсером (Apache POI, PDFBox); прогоняйте через Content Disarm & Reconstruction (CDR), если модель угроз оправдывает.
- Для архивов (ZIP, tar) — считайте опасными. Здесь живёт куча классов атак: zip-slip path traversal, zip-bomb, глубина вложенных архивов, symlink-записи. Если фича допускает архивы, требуйте:
  - Путь извлечения остаётся под целевым каталогом (проверять после join).
  - Суммарный uncompressed-размер *и* коэффициент сжатия ограничены (10-КБ файл, разворачивающийся в 10 ГБ, — zip-bomb).
  - Записи с абсолютными путями или `..` — отклонять.
  - Записи-симлинки и device-node — отклонять сразу.
- Когда модель угроз допускает задержку, прогоняйте аплоады через AV / sandbox-анализатор до появления у других пользователей.

## Хранилище

- Храните загруженные файлы **вне web-root**. Приложение раздаёт их через контролируемый download-эндпоинт, а не прямой URL.
- В идеале храните на отдельном сервере или в object storage (S3 и аналоги). Если локально — выставляйте права так, чтобы процесс веб-сервера не мог исполнять файлы в каталоге аплоада (нет `+x`; отключён `Options ExecCGI`; `php_admin_flag engine off` в `.htaccess` или эквивалент).
- Используйте application-layer mapping: `GET /files/<opaque-id>` резолвится в `<uuid>` на диске. Пользователь никогда не видит имён хранилища.
- Для БД: хранение мелких файлов в БД разумно в отдельных кейсах (аттачи, жёстко связанные со строкой), но обычно плохо масштабируется и усложняет бэкап/рестор. Объектное хранилище — хороший дефолт.

## Контроль доступа

- Для загрузки обязательна аутентификация.
- Авторизация определяет, кто какой тип файла может загружать и в какую «папку» / контекст. См. `rule-authorization-access-control.md`.
- **Каждое скачивание** проверяет авторизацию. Не полагайтесь на непрозрачность имени — это security through obscurity. Непрозрачное имя плюс authz-проверка — правильно.
- Права ФС — по принципу наименьших: веб-воркер не может писать вне upload-дерева; само upload-дерево не читается никем другим.

## Лимиты размера и rate

- Жёсткий размер на файл (на уровне веб-сервера и приложения; оба, не только один).
- Aggregate-лимиты на запрос и на пользователя (размер / количество).
- Лимит размера после декомпрессии для сжатых аплоадов (см. архивы).
- Для скачиваний — rate-limit для защиты от исчерпания канала и скрейпа.

## Безопасная выдача

- Ставьте `Content-Disposition: attachment; filename="..."`, чтобы браузер трактовал ответ как скачивание, а не рендер. Это убивает большинство багов «загруженный HTML становится stored XSS».
- Ставьте `X-Content-Type-Options: nosniff`, чтобы браузер не «угадывал» MIME.
- Ставьте `Content-Security-Policy: default-src 'none'; frame-ancestors 'none'` на ответ, чтобы ограничить всё, что всё-таки рендерится.
- По возможности раздавайте из **другого origin** чем приложение. Даже если вредоносный SVG отрендерится, это будет в одноразовом sandbox-origin-е, не там, где живут сессионные cookie пользователя.

## Дополнительные контроли

- CSRF-защита на upload-эндпоинте — как у любого state-changing запроса (`rule-client-web-browser.md`).
- Логируйте каждую загрузку: кто, когда, размер, заявленный тип, sniffed-тип, полученный opaque-ID. Дашборды на этом логе ловят аномалии.
- Держите парсинг-библиотеки свежими (`rule-supply-chain-dependencies.md`). Библиотеки изображений и документов — типичный источник CVE.
- Где это уместно, предусматривайте механизм пользовательских жалоб на контент.

## Чек-лист реализации

1. Проверки расширения, content-type и magic-byte прошли и все согласны с allow-list.
2. Имена файлов генерятся на сервере; оригинальные — только как метаданные, если вообще.
3. Изображения перекодируются; документы валидируются; архивы проверяются по размеру и путям.
4. Хранилище вне web-root; веб-сервер не может исполнять файлы в upload-дереве.
5. Аутентифицированный upload, авторизованный download, прямые URL не экспонированы.
6. Лимиты размера на нескольких уровнях; ограничение коэффициента сжатия для архивов.
7. `Content-Disposition: attachment`, `X-Content-Type-Options: nosniff` и в идеале sandbox-origin для выдачи.
8. CSRF-защита на upload; структурное логирование событий.
