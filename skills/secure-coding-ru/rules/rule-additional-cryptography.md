# Прикладная криптография — TLS, ключи, данные at rest

Это правило покрывает *применение* криптопримитивов. Выбор самих примитивов (AES-GCM vs ChaCha20, Ed25519 vs ECDSA, параметры Argon2id и т. п.) — в `always-crypto-algorithms.md`.

## Данные at rest

- Шифруйте чувствительные данные. Порог здесь ниже, чем многие помнят: «чувствительное» — это всё, что может идентифицировать пользователя, раскрыть бизнес-отношение или помочь последующей атаке.
- Минимизируйте то, что хранится. Самое сильное шифрование — не хранить значение вовсе.
- Для идентификаторов, над которыми не нужно считать (номера карт, банковские счета, ID для сравнения на равенство) — предпочитайте **токенизацию**.
- Используйте **аутентифицированное шифрование** (AES-GCM или ChaCha20-Poly1305). Неаутентифицируемые режимы позволяют атакующему переворачивать биты в шифртексте незаметно.
- **Управление IV / nonce**: уникальны на сообщение; никогда не переиспользовать. Для AES-GCM — 96-битный случайный nonce или счётчик, если гарантируете отсутствие переиспользования между процессами. Одно повторение nonce под одним ключом в AES-GCM катастрофично — раскрывает XOR двух plaintext-ов *и* позволяет подделывать шифртекст.
- **Соли** уникальны на запись и хранятся рядом с шифртекстом. Секретом быть не обязаны, но переиспользоваться не должны.

### Бэкапы

- Бэкапы шифруются ключом, отличным от продового. Компрометация live-сервиса не должна расшифровывать все исторические снапшоты.
- Пути восстановления проверяются — непроверенное восстановление не бэкап.
- Retention и доступ управляются политикой; аудитируйте логи доступа при событиях восстановления.

## Управление ключами

- Генерируйте ключи внутри **валидированного модуля** — HSM, KMS или secure enclave платформы. Если ключ генерится в коде приложения, он генерится с той энтропией, что досталась процессу.
- Никогда не выводите долгоживущий ключ из пароля как единственного источника энтропии.
- Разделяйте ключи по назначению: шифрование, подпись, wrap, аутентификация. Компрометация одного не должна ронять остальные.
- **Envelope encryption**: данные шифруются data-encryption key (DEK); DEK зашифрован key-encryption key (KEK), живущим в KMS. Шифрованные данные + wrapped DEK лежат вместе; KEK используется только для unwrap.
- **Ротация** по расписанию *и* по любому сигналу компрометации. Система должна уметь ротацию без downtime — план миграции готовим до того, как он понадобится.
- Доступ к ключам аудитируется. Кто, что, когда, откуда unwrap-нул.

## Настройка TLS

### Версии протокола

- Включайте TLS 1.3. TLS 1.2 — только если конкретный legacy-клиент этого требует, с задокументированным исключением.
- Отключайте TLS 1.0, TLS 1.1 и все SSL-версии. В современных серверах они отключены по умолчанию; проверяйте конфиг.
- Включайте `TLS_FALLBACK_SCSV`, чтобы защититься от downgrade в смешанных парках клиентов.

### Cipher suites

- Предпочитайте AEAD-cipher suites: TLS 1.3 их и требует; для TLS 1.2 — `TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384`, `TLS_ECDHE_RSA_WITH_CHACHA20_POLY1305_SHA256` и аналоги.
- Отключайте NULL, EXPORT, анонимные, RC4, 3DES и SEED.
- Отключайте TLS-сжатие — атака CRIME.

### Группы обмена ключами

- Предпочитайте `x25519`, затем `secp256r1` (P-256). `secp384r1` — только если требует политика.
- Если FFDHE допустим, используйте именованные группы из RFC 7919 (`ffdhe2048` — минимум).
- Про гибридные пост-квантовые группы — см. `always-crypto-algorithms.md`.

### Сертификаты

- Размер ключа: RSA ≥ 2048 (предпочтительно 3072) или EC на P-256 / P-384. Инспекция — в `always-certificate-hygiene.md`.
- Подпись: SHA-256 или сильнее.
- Точное совпадение `CN` / `SAN` с используемыми хостами; избегайте wildcard-сертификатов, покрывающих не связанные хосты.
- OCSP stapling включён, чтобы не платить перформансом и приватностью за клиентский OCSP.
- Жизненный цикл — ротируйте с перекрытием, мониторьте срок, имейте план отката.

### Прикладной уровень

- HTTPS сайтово. HTTP редиректится на HTTPS.
- Никакого mixed content — одна HTTP-субресурс на HTTPS-странице ломает весь origin.
- Все cookie несут атрибут `Secure`.

## HSTS

`Strict-Transport-Security` говорит браузерам отказываться от HTTP для хоста (и, опционально, всех поддоменов) на срок, заданный политикой.

Постадийный rollout:

1. **Тест**: `max-age=86400` (1 день), с `includeSubDomains`, если поддомены реально только HTTPS. Мониторьте.
2. **Продакшен**: минимум `max-age=31536000` (1 год). `includeSubDomains` добавляйте, когда каждый поддомен корректно отдаёт HTTPS.
3. **Preload**: подавайте заявку в HSTS preload, когда политика проработала в проде без проблем несколько месяцев. Помните: preload фактически постоянен — удаление проходит неделями.

HSTS никогда не отправляйте по HTTP; он игнорируется и говорит о путанице.

## Pinning сертификатов

Пинните аккуратно. Это отличный шанс отстрелить себе ногу.

- HPKP **не использовать** (устарел; браузеры удалили).
- Pinning имеет смысл для **контролируемых** клиентов — мобильных приложений, киосков, встраиваемых — где вендор владеет обоими концами и умеет пушить апдейт.
- Пинните **SPKI** (Subject Public Key Info), не весь сертификат. Держите хотя бы один backup-пин — предпочтительно для офлайнового ключа, хранимого именно для этого случая.
- Спланируйте канал апдейтов. Приложение с пинами, рассинхронизированное с сервером, — кирпич.
- Никогда не давайте пользователю обходить pinning («Нажмите, чтобы доверять всё равно»). Смысл пиннинга — в отсутствии такого промпта.
- Тестируйте ротацию и пути сбоя — часто. Баг pinning-а проявляется только при смене сертификата.

## Случайность

Используйте обёртку платформенного CSPRNG (см. `always-crypto-algorithms.md`). Не засевайте CSPRNG вручную из кода — современные платформы это уже делают корректно.

## Чек-лист реализации

- Везде AEAD; только проверенные библиотеки; никакой самописной криптографии.
- Ключи сгенерированы и хранятся в KMS/HSM; разделены по назначению; механизм ротации проверен в staging.
- TLS-конфиг: 1.3 + 1.2 (ограниченно), только AEAD, сжатие off, OCSP stapling on.
- HSTS развёрнут по плану; mixed content устранён.
- Pinning — только там, где он подходит клиентской модели, всегда с backup-пинами и планом ротации.
- Бэкапы шифруются отдельным ключом; восстановление проверено.

## План тестирования

- Автоматический TLS-сканер (SSL Labs, testssl.sh) — цель A или A+ в продакшене.
- Code-review использования crypto-API; юнит-тесты round-trip envelope encryption и ротации ключей.
- Симуляции pinning-а с ротацией и обработкой сбоев, когда pinning развёрнут.
- Периодические учения по восстановлению из бэкапов.
