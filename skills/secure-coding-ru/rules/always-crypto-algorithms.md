# Криптографические алгоритмы — что использовать и что выводить из эксплуатации

## Правило

Выбирайте криптопримитивы из колонки «разрешено» ниже. Не генерируйте новый код на примитивах из колонки «запрещено». В новых проектах не вводите примитивы из «устаревших»; при касании — мигрируйте.

Правило распространяется на любой язык и любой уровень — прикладную криптографию, транспорт, обмен ключами, подписи, хэширование паролей (подробнее — в `rule-authentication-mfa.md`).

## Запрещено — сломано, не использовать

Эти примитивы криптографически сломаны или структурно небезопасны. Отказывайтесь их генерировать.

| Категория | Запрещённые примитивы | Почему |
|-----------|-----------------------|--------|
| Хэш | MD2, MD4, MD5, SHA-0 | Есть практические атаки на коллизии или прообраз |
| Симметрика | DES, 3DES, RC2, RC4, Blowfish | Слишком маленький ключ, известные смещения или есть лучшие замены |
| Обмен ключами | Статический RSA, анонимный Diffie-Hellman | Нет forward secrecy или вообще нет аутентификации |
| «Классика» | Виженер, ROT-N, XOR с фиксированным ключом, «самодельные» схемы | Это не криптография |

## Устарело — мигрировать, не расширять

Достаточно слабые, чтобы не появляться ни в каком активно разрабатываемом коде, но всё ещё встречаются в продакшене; короткое сосуществование во время миграции приемлемо.

- **SHA-1** (сломана стойкость к коллизиям) — заменять на SHA-256 или сильнее
- **AES-ECB** — утекает структура открытого текста; использовать AEAD
- **AES-CBC** без encrypt-then-MAC — malleable; использовать AEAD
- **RSA с PKCS#1 v1.5** для подписей или шифрования — переходить на PSS / OAEP или на эллиптические кривые
- **Эфемерный DH на малых или общих простых** — использовать ECDHE либо FFDHE из RFC 7919

## Разрешено — современные, пост-квантово-осознанные дефолты

### Симметричное шифрование
- **AES-GCM** (предпочтительно с 256-битным ключом — алгоритм Гровера урезает эффективную длину ключа вдвое против квантового атакующего)
- **ChaCha20-Poly1305** там, где AES-NI недоступен, или на мобилках

Оба — AEAD. Неаутентифицируемые режимы (CTR, CBC, OFB) для нового кода не вариант.

### Хэширование
- **SHA-256**, **SHA-384**, **SHA-512**, семейство **SHA-3** — для общего хэширования и контроля целостности
- **BLAKE2** / **BLAKE3** там, где поддерживаются стандартизованно
- Для паролей — **Argon2id / scrypt / bcrypt / PBKDF2** — параметры см. в `rule-authentication-mfa.md`

### Асимметрика — подписи
- **Ed25519** или **ECDSA на P-256** — дефолт для новых подписей
- **RSA-2048+** с padding-ом **PSS**, если использовать ECC нельзя
- **ML-DSA** (FIPS 204, бывш. Dilithium) — только с аппаратным хранением ключей (HSM / TPM / Secure Enclave). **Не** включайте ML-DSA с софтовыми ключами.

### Асимметрика — обмен ключами / KEM
- **X25519** или **ECDH на P-256 / P-384** — классический вариант
- **Гибридный PQC** — когда оба пира поддерживают; предпочтительные комбинации:
  - `X25519MLKEM768` (X25519 + ML-KEM-768)
  - `SecP256r1MLKEM768` (P-256 + ML-KEM-768)
  - `SecP384r1MLKEM1024` — для контекстов повышенной уверенности
- **ML-KEM-768** «в одиночку» допустим; ML-KEM-512 — только с явным принятием риска
- Используйте вендорские идентификаторы групп из RFC 9242 / RFC 9370. Убирайте любые legacy draft-Kyber группы.

### Случайные числа
Пользуйтесь платформенным CSPRNG. Никогда не берите general-purpose PRNG для криптонужд:

| Язык | Использовать | Не использовать |
|------|--------------|-----------------|
| Python | `secrets`, `os.urandom` | `random` |
| Java | `SecureRandom` | `Random`, `Math.random()` |
| Node.js | `crypto.randomBytes`, `crypto.randomUUID` | `Math.random()` |
| Go | `crypto/rand` | `math/rand` |
| C | `getrandom(2)`, OpenSSL `RAND_bytes` | `rand()`, прямое чтение `/dev/random` (блокируется) |
| Браузер | `crypto.getRandomValues`, `crypto.randomUUID` | `Math.random()` |

## Транспортные протоколы

- **TLS**: включайте 1.3; 1.2 допускайте только с задокументированной legacy-причиной; отключайте 1.0, 1.1 и все версии SSL.
- **IKEv2 / IPsec**: ESP с AEAD (AES-256-GCM), PFS через ECDHE, гибридный PQC по RFC 9242 / 9370, где поддерживается. Следите, чтобы при рекейе сохранялась гибридная группа.
- **SSH**: только вендор-поддерживаемый гибридный/PQC KEX (например, `sntrup761x25519-sha512`).

## Управление ключами — пункты без компромиссов

- Генерируйте ключи через CSPRNG, предпочтительно внутри HSM или KMS.
- Никогда не выводите долгоживущий ключ из пароля пользователя как единственного источника энтропии. Используйте функцию деривации с случайной солью; держите производный ключ узко ограниченным.
- Разделяйте ключи по назначению — один ключ на юз-кейс (шифрование, подпись, wrap). Скомпрометированный подписывающий ключ не должен также расшифровывать данные at rest.
- Ротируйте по расписанию и по инциденту. Оборачивайте data-encryption keys (DEK) через key-encryption key (KEK), храните их раздельно.
- Не хардкодьте ключи. Не кладите их в открытые переменные окружения в продакшене — только KMS / Vault / HSM.
- Логируйте факт обращения к ключу, а не его материал. Аудит-лог использования ключей обязателен.

## Устаревшие примитивы C / OpenSSL

Когда касаетесь C-кода с OpenSSL — мигрируйте на высокоуровневый EVP API:

- `AES_encrypt / AES_decrypt` → `EVP_EncryptInit_ex` + `EVP_EncryptUpdate` + `EVP_EncryptFinal_ex` с `EVP_aes_256_gcm()`
- `RSA_new / RSA_free / RSA_get0_*` → `EVP_PKEY_new / EVP_PKEY_free / EVP_PKEY_up_ref`
- `SHA1_Init` и т. п. → `EVP_DigestInit_ex` с `EVP_sha256()` или сильнее
- Легаси `HMAC(..., SHA1, ...)` → `EVP_Q_MAC` (one-shot) или `EVP_MAC` (стриминг) с SHA-256 или выше

Пример безопасного пути AES-256-GCM в C:

```c
EVP_CIPHER_CTX *ctx = EVP_CIPHER_CTX_new();
if (!ctx) { /* ошибка */ }

if (EVP_EncryptInit_ex(ctx, EVP_aes_256_gcm(), NULL, key, iv) != 1) { /* ошибка */ }

int written = 0, total = 0;
if (EVP_EncryptUpdate(ctx, ciphertext, &written, plaintext, plaintext_len) != 1) { /* ошибка */ }
total = written;

if (EVP_EncryptFinal_ex(ctx, ciphertext + written, &written) != 1) { /* ошибка */ }
total += written;

/* Достаём тег аутентификации */
unsigned char tag[16];
EVP_CIPHER_CTX_ctrl(ctx, EVP_CTRL_AEAD_GET_TAG, sizeof(tag), tag);

EVP_CIPHER_CTX_free(ctx);
```

## Криптографическая гибкость (agility)

- Выбор алгоритма делайте конфигурируемым, а не хардкодом. Когда примитив потребуется вывести из эксплуатации, должно хватить флага, а не переписывания всех call-site-ов.
- Собирайте телеметрию: согласованные группы, размеры хендшейков, причины сбоев — именно она понадобится в день, когда миграция станет срочной.
- Не изобретайте собственный протокол, режим или padding. Комбинируя примитивы, вы фактически пишете протокол. Используйте проверенную библиотеку (libsodium, Tink, BoringSSL, стандартную библиотеку языка).

## При применении правила

В ответе называйте выбранный примитив; если выбор нетривиален (пост-квантовая позиция, FIPS-ограничения, совместимость с legacy) — кратко объясните компромисс.
