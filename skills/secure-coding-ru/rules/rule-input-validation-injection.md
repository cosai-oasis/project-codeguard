# Валидация ввода и защита от инъекций

## Суть

Любая инъекция устроена одинаково: строку, произведённую недоверенной стороной, передают интерпретатору (SQL-движку, шеллу, LDAP-серверу, шаблонизатору, JS-рантайму), который её часть трактует как код, а не данные. Защита — двухслойная:

1. **Валидация на границе доверия** — определите, какую форму ввод может принимать, всё остальное отбрасывайте *до* того, как оно дойдёт до любого интерпретатора.
2. **API, которые держат данные вне кодового канала** — параметризованные запросы, структурированное исполнение команд, allow-listed токены. Экранирование — фолбэк, а не основная защита.

Оба слоя важны. Валидация не спасёт непараметризованную SQL-строку; параметризация не остановит вызывающего, передающего 100-мегабайтный payload.

## Плейбук валидации

### Positive (allow-list) валидация

Для каждого недоверенного поля определяйте, что считается валидным, и отбрасывайте всё остальное. Не пытайтесь перечислить «что нельзя» — такой список всегда неполон.

- **Синтаксика**: тип, формат (regex с якорями `^…$`), числовой диапазон, длина строки, принадлежность перечислению.
- **Семантика**: бизнес-правила, которые синтаксисом не выразить (`start ≤ end`, итоги заказа совпадают со строками позиций, пользователь владеет упомянутым объектом).
- **Нормализация перед валидацией**: канонизируйте Unicode (NFKC), декодируйте процент-кодирование, разворачивайте эквивалентные представления. Валидируйте *нормализованное* значение, затем используйте его везде одинаково.

### Свободный текст

Когда ввод действительно текст:
- Задайте допустимый класс символов (по Unicode-категориям, а не только ASCII).
- Проставьте жёсткую границу длины.
- Нормализуйте Unicode.
- Помните: «свободный текст» не освобождает от XSS-экранирования при дальнейшем рендере.

### На что смотреть в regex

- Всегда якорьте (`^…$`) — regex без якорей почти никогда не то, что вы хотите.
- Бойтесь ReDoS: вложенные квантификаторы (`(a+)+`), альтернация с общими префиксами, `.*` перед обязательным совпадением — типичные триггеры. Предпочитайте парсерные валидаторы либо атомарные/possessive-группы, если движок их поддерживает.

### Файлы

Файлы — тоже ввод. Валидируйте по содержимому (magic-bytes), по жёсткому лимиту размера и по расширению. Генерируйте серверное имя файла, а не доверяйте клиентскому. Полный чек-лист — в `rule-file-upload-handling.md`.

## SQL-инъекции — параметризация, а не экранирование

Дефолт — параметризованные запросы / bound-переменные для 100% обращений к данным. Конкатенация пользовательского ввода в SQL недопустима.

Java (JDBC):

```java
String query = "SELECT account_balance FROM user_data WHERE user_name = ?";
try (PreparedStatement pstmt = connection.prepareStatement(query)) {
    pstmt.setString(1, request.getParameter("customerName"));
    try (ResultSet rs = pstmt.executeQuery()) {
        // …
    }
}
```

Python (DB-API 2.0):

```python
cur.execute("SELECT balance FROM accounts WHERE user_id = %s", (user_id,))
```

Go (database/sql):

```go
row := db.QueryRowContext(ctx, "SELECT balance FROM accounts WHERE user_id = $1", userID)
```

Node.js (pg):

```js
const { rows } = await client.query(
  'SELECT balance FROM accounts WHERE user_id = $1',
  [userID]
);
```

### Динамические идентификаторы (имена колонок/таблиц)

Плейсхолдеры связывают *значения*, а не идентификаторы. Если имя колонки или таблицы должно меняться, проверяйте его по явному allow-list известных строк. Никогда не собирайте идентификатор из пользовательского ввода конкатенацией.

### Хранимые процедуры

Хранимка сама по себе не защищает от SQL-инъекции. Если внутри она собирает динамический SQL конкатенацией — дыра по-прежнему есть. Параметризуйте и внутри процедуры.

### Привилегии

У прикладной DB-учётки не должно быть прав на DDL, superuser/root или доступа к таблицам вне своей зоны. SQL-инъекция несравнимо менее опасна, когда скомпрометированный аккаунт может читать только три таблицы.

## SOQL / SOSL (Salesforce Apex)

У SOQL и SOSL нет DDL/DML-поверхности SQL, но ими всё ещё можно манипулировать для обхода фильтров и экфильтрации — а если результат пойдёт дальше в DML, радиус поражения растёт.

- Предпочитайте статические запросы с bind-переменными:
  - `[SELECT Id FROM Account WHERE Name = :userInput]`
  - `FIND :term IN ALL FIELDS RETURNING Account`
- Динамический SOQL → `Database.queryWithBinds()`; динамический SOSL → `Search.query()`.
- Любой динамический идентификатор (поле, объект, order-by) проходите через allow-list.
- `String.escapeSingleQuotes()` — только когда конкатенация действительно неизбежна.
- Обеспечивайте CRUD/FLS через `WITH USER_MODE` или `WITH SECURITY_ENFORCED` (что-то одно, не совмещать). Используйте `with sharing` или user-mode DML. Перед DML прогоняйте `Security.stripInaccessible()`, чтобы срезать недоступные вызывающему поля.

## LDAP-инъекции

В LDAP два контекста инъекций, требующих разного экранирования:

- **Distinguished Name (DN)** — экранируйте `\  #  +  <  >  ,  ;  "  =` и ведущие/завершающие пробелы.
- **Фильтр** — экранируйте `*  (  )  \  NUL`.

Валидируйте allow-list-ом до построения запроса, используйте библиотеку, дающую корректное DN/filter-кодирование (например, `javax.naming.ldap.LdapName`, Python-овский `ldap3.utils.dn.escape_rdn`). Привязывайтесь как наименее привилегированная рабочая учётка.

## Инъекции в OS-команды

Дефолтный ответ: не вызывайте шелл. В стандартной библиотеке почти всегда есть функция, делающая ровно то, что делает shell-команда.

Если subprocess действительно нужен:

- Используйте структурированное исполнение, где команда и аргументы лежат в разных слотах, — не одной склеенной строкой. `ProcessBuilder` в Java, `subprocess.run([...], shell=False)` в Python, `exec.Command(name, args...)` в Go.
- **Не** вызывайте шелл (`/bin/sh -c`, `cmd.exe /c`). В момент, когда шелл оказывается в цепочке, метасимволы становятся активны.
- Allow-list-ом фиксируйте саму команду и валидируйте каждый аргумент по allow-list regex-у (никаких `&  |  ;  $  >  <  \` ``  !  "  '  (  )`, никаких пробелов, если они не задуманы).
- Где поддерживается — используйте `--`, чтобы отделить опции от позиционных аргументов и защититься от флаг-инъекции.

Java (ProcessBuilder, безопасный шаблон):

```java
ProcessBuilder pb = new ProcessBuilder("/usr/bin/trusted_tool", "--mode", modeArg, "--", fileArg);
pb.directory(new File("/var/app/work"));
pb.redirectErrorStream(true);
Process p = pb.start();
```

Python:

```python
subprocess.run(
    ["/usr/bin/trusted_tool", "--mode", mode_arg, "--", file_arg],
    shell=False,
    check=True,
    timeout=30,
)
```

## Инъекции в шаблонизаторы / язык выражений

Многие шаблонизаторы (Jinja2, Freemarker, Velocity, Handlebars, Twig, ERB) с удовольствием исполнят код, если им отдать недоверенный шаблон. Относитесь к ним как к опасным:

- Никогда не рендерьте шаблон, исходная строка которого пришла от пользователя.
- Если пользовательский ввод надо подставить, передавайте его переменной в render-контекст, а не конкатенируйте в тело шаблона.
- Где можно — отключайте рантайм-eval (песочница, `StrictUndefined` и т. п.).

## Prototype pollution (JavaScript / TypeScript)

Цепочка прототипов в JS означает, что присваивания в `__proto__`, `constructor.prototype` и `prototype` на общих объектах «протекают» в каждый соседний объект.

Шаблоны защиты:

- Для словарей предпочитайте `new Map()` / `new Set()` обычным объектам.
- Когда словарём обязан быть объект, создавайте его без прототипа: `Object.create(null)` или `{ __proto__: null }`.
- Замораживайте/запечатывайте то, что не должно меняться (`Object.freeze`, `Object.seal`).
- Избегайте непроверенных deep-merge-утилит. Если мёржите — валидируйте ключи allow-list-ом и явно блокируйте `__proto__`, `constructor`, `prototype`.
- В уплотнённой среде запускайте Node.js с `--disable-proto=delete` как эшелонированной защитой.

## NoSQL-инъекции (MongoDB и т. п.)

BSON-объект, распарсенный из JSON, — *всё ещё* структурированный ввод. Если клиент может отправить `{"$ne": null}` там, где ожидалась строка, значит у вас operator injection.

- Валидируйте, что поля фильтра имеют ожидаемые скалярные типы, перед передачей в драйвер.
- В ORM там, где возможно, отключайте парсинг операторов (например, `sanitizeFilter` у Mongoose).
- Используйте параметризованные query-builder-ы вместо ручного построения filter-объектов.

## XPath / XQuery-инъекции

Параметризуйте через API (`javax.xml.xpath.XPath.setXPathVariableResolver`, `lxml.etree.XPath(expr, variable_dict)`), не конкатенируйте ввод в выражение.

## Кэширование и транспорт

- Ответы с валидированными, но чувствительными данными должны идти с `Cache-Control: no-store`. Валидатор ничем не помогает, если общий кэш уже отдал чужую строку.
- Включайте HTTPS на каждом эндпоинте, принимающем недоверенный ввод, — on-path-атакующий успеет переписать вход до того, как валидатор его увидит.

## Чек-лист реализации

- Для каждого типа ввода есть центральный валидатор (и он реально используется, а не обходится для «простых» эндпоинтов).
- 100% параметризация для SQL и SOQL; динамические идентификаторы идут через allow-list.
- DN- и filter-кодирование в LDAP делает библиотечный вызов, а не ручная замена строк.
- Шелл с недоверенным вводом не запускается; если используется subprocess — команда и аргументы структурированы и allow-list-ятся.
- JS-словари создаются безопасными конструкторами; пути прототипов заблокированы в любой merge-утилите.
- Шаблонизаторы никогда не рендерят пользовательские шаблоны.
- Любая загрузка файлов проходит по чек-листу `rule-file-upload-handling.md`.

## План тестирования

- Статический скан на конкатенацию строк в sink-ах запросов/команд (`Statement.executeQuery(".."+x)`, `exec.Command("sh", "-c", ...)` и т. п.).
- Фаззинг классическими payload-ами: `' OR 1=1--`, `${jndi:ldap:...}`, `$(id)`, `{{7*7}}`, `__proto__`.
- Юнит-тесты на пограничные случаи валидатора — граничные длины, Unicode-омоглифы, смешанные кодировки.
- Негативные тесты на каждый заблокированный ключ прототипа и каждую отклонённую форму файла.
