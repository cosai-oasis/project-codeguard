# Сессии и cookie

## Цели проектирования

Сессия должна быть неподделываемой, угоняемой только через компрометацию транспорта или клиента, автоматически инвалидируемой при смене привилегий и аудируемой — но без утечки чувствительного материала в логи.

## Свойства session ID

- **Источник** — CSPRNG. ≥ 64 бит энтропии — нижняя граница; ≥ 128 бит — разумный дефолт. 256 бит генерировать дёшево и тривиально достаточно.
- **Непрозрачность** — ID не кодирует смысл. Никаких user ID, роли, tenant-а, timestamp. Это живёт в серверном состоянии по ключу ID.
- **Происхождение** — session ID создаёт только сервер. Входящий ID, которого сервер не выпускал, должен отклоняться, а не приниматься и эскалироваться.
- **Имя cookie** — короткое нейтральное (`id`, `sid`) нормально; дефолты фреймворков (`PHPSESSID`, `JSESSIONID`, `ASP.NET_SessionId`) сигнализируют атакующему о стеке. Либо переименуйте, либо хотя бы не полагайтесь на само имя.
- **Состояние на сервере** — все данные сессии живут на сервере. Не гоняйте чувствительные атрибуты через клиент. Серверные session-хранилища с чувствительными данными шифруются at rest.

## Флаги cookie

Для каждого session cookie:

```
Set-Cookie: id=<opaque>; Secure; HttpOnly; SameSite=Strict; Path=/
```

- **`Secure`** — cookie никогда не уходит по открытому HTTP. В связке с HSTS HTTP недостижим в принципе.
- **`HttpOnly`** — cookie недоступен из JavaScript. Защищает сессию от любого XSS, который иначе читал бы `document.cookie`.
- **`SameSite=Strict`** — cookie не отправляется при кросс-сайт запросах, включая верхнеуровневые навигации. `Lax` — если верхнеуровневый GET по внешней ссылке обязан нести сессию; `Strict` предпочтительнее для чувствительных путей.
- **`Path` / `Domain`** — сужайте скоуп cookie максимально. Избегайте кросс-поддоменных cookie, если они вам реально не нужны.
- **Предпочитайте непостоянные** session cookie (без `Expires` / `Max-Age`) — они умирают с закрытием браузера.
- Рассмотрите префикс **`__Host-`**, чтобы принудительно выставить `Secure`, `Path=/` и без атрибута `Domain` — небольшая, но реальная защита от subdomain takeover.

## Жизненный цикл сессии

1. **Создание**: только сервер создаёт сессии. Ничему от клиента про ID не доверяйте.
2. **Аутентификация**: ротируйте session ID при успешном логине. Инвалидируйте пре-auth ID. Это главная защита от session fixation.
3. **Смена привилегий**: ротируйте заново при смене пароля, enroll/remove MFA, эскалации роли или любом действии, расширяющем права.
4. **Logout**: хендлер `/logout` инвалидирует серверную запись сессии *и* очищает cookie на клиенте. Нужны оба шага — любой в одиночку оставляет дыру.
5. **Уничтожение**: idle и absolute таймауты — на сервере. См. ниже.

Если фреймворк предпочитает разные имена cookie для pre-auth и post-auth — используйте этот паттерн; он упрощает аудит шага инвалидации.

## Таймауты

| Чувствительность | Idle timeout | Absolute timeout |
|------------------|--------------|------------------|
| Высокая (банкинг, админка, медицина) | 2–5 минут | 4 часа |
| Средняя | 15–30 минут | 4–8 часов |
| Низкая | До 1 часа | 12–24 часа |

Оба таймаута должны контролироваться на сервере. Клиентский таймер, который говорит пользователю «вы разлогинены», не уничтожая сессию на сервере, — декорация, а не контроль.

Кнопка logout должна быть видна на каждой аутентифицированной странице и работать без JavaScript.

## Транспорт и кэширование

- Только HTTPS на всём пути сессии. Один HTTP-hop — и cookie утекла. Если приложение обязано работать по HTTP для части пользователей, через HTTP аутентификации быть не должно.
- Включайте HSTS (`Strict-Transport-Security`) сайтово.
- Страницы, где в теле ответа есть session-идентификаторы (или чувствительные пользовательские данные), несут `Cache-Control: no-store`. Общие кэши здесь — реальная угроза.

## Детекция угона и ответ

Session-токен можно вынести через уязвимости браузера, вредоносные расширения, физический доступ. Сервер способен детектить *подозрительное* повторное использование по фингерпринту окружения.

Фиксируйте при установлении:

- IP-адрес (лучше по подсети, чем точный — много легитимных пользователей роумят),
- user agent,
- `Accept-Language`,
- доступные `sec-ch-ua`-хинты,
- гео-бакет.

Сравнивайте на каждом запросе. Небольшой дрифт допускайте; на крупные скачки реагируйте.

| Уровень риска | Триггеры | Реакция |
|--------------|----------|---------|
| Низкий | Смена минорной версии UA | Лог, продолжаем |
| Средний | Смена ASN, смена major-версии UA | Challenge (re-verify MFA), ротация session ID |
| Высокий | Смена страны, одновременные сессии из далёких мест, известный-плохой ASN | Принудительная re-auth, ротация session ID, алёрт пользователю |

На каждом подозрительном событии ротируйте session ID; угонщик со старым ID остаётся без логина.

## Клиентское хранилище

Session-токенам **не** место в `localStorage` или `sessionStorage`. Любой XSS их заберёт. Cookie с `HttpOnly` — единственное разумное место для session-материала.

Если архитектуре нужен токен, доступный JS (скажем, bearer-токен для API), — признайте, что XSS его получит, и:

- Минимизируйте время жизни токена (минуты, не дни).
- Привязывайте токен к дополнительному клиентскому контексту, чтобы голая кража не срабатывала.
- Подумайте об изоляции хранилища в Web Worker с узким message-API в основной поток.

## Сценарии с несколькими cookie

Если приложение использует несколько cookie, вместе описывающих состояние сессии (например, access- и CSRF-cookie):

- Проверяйте их связь друг с другом на каждом запросе; не доверяйте им независимо.
- Не давайте им одинаковое имя с разным path/domain — поведение cookie jar в браузерах неконсистентно.

## Мониторинг

- Логируйте события жизненного цикла сессии (created, rotated, terminated, detected-drift) с **соль+хэш** session ID, не сырой ID. Вам нужно корреляция между записями без превращения лога в дамп кредов.
- Дашборды: брутфорс session ID (высокий rate по одному ID), аномальная параллельная работа, неудачные события жизненного цикла.

## Чек-лист реализации

1. Session ID из CSPRNG, непрозрачные, ≥ 64 бит энтропии, выпускаемые сервером.
2. Флаги cookie: `Secure`, `HttpOnly`, `SameSite`, узкие `Path` / `Domain`. Префикс `__Host-` где возможно.
3. HTTPS + HSTS везде. Никакого mixed content.
4. Ротация session ID при аутентификации и при смене привилегий. Инвалидация предыдущего.
5. Idle и absolute таймауты — на сервере. Хендлер logout уничтожает серверное состояние и чистит cookie.
6. `Cache-Control: no-store` на чувствительных ответах.
7. Серверный фингерпринт с риск-ответом на дрифт.
8. Session-токен — никогда в web-хранилище. Дефолты фреймворка усилены, а не приняты как есть.
