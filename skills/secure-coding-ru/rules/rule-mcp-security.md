# Безопасность MCP (Model Context Protocol)

MCP даёт LLM контролируемый доступ к tools, resources и prompts. Этот доступ и есть атакующая поверхность: вредоносный MCP-сервер может эксфильтрировать данные, скомпрометированного клиента можно заставить вызвать опасный tool, а prompt-injection ходит через оба. Контроли ниже — базовый уровень для любого продового MCP-развёртывания.

## Workload identity

- Используйте криптографический workload identity — SPIFFE / SPIRE как референс для мультиклауда; эквивалент облачного вендора (AWS IAM Roles for Service Accounts, GCP Workload Identity Federation, Azure Managed Identity) для одиночного облака.
- Идентичности короткоживущие (SVID с TTL в минутах), ротация автоматическая.
- Одна идентичность на одну логическую службу. Не шарьте идентичности между серверами с разным профилем прав.

## Санитизация ввода и данных

Считайте **любой** вход MCP недоверенным, независимо от источника:

- Пользовательские промпты, схемы tool-ов, содержимое ресурсов, метаданные — любой из них может нести prompt-injection, который downstream-LLM исполнит.
- Применяйте allow-list-валидацию на каждой границе доверия.
- Канонизируйте пути к файлам до ФС-операций; отклоняйте всё, что выходит за разрешённый корень.
- Параметризуйте запросы к БД (`rule-input-validation-injection.md`).
- До возврата данных применяйте контекстное экранирование (SQL-, shell-, HTML-контексты требуют разных экранирований).

### Гигиена вывода tool-а

- Возвращайте только те поля, что реально нужны вызывающему. Дамп всей записи из внутренней БД — подарок скомпрометированной модели.
- Редактируйте PII до того, как данные покинут сервер.
- Ставьте жёсткий размер ответа.

### Prompt-injection

- Разверните слой детекции prompt-injection (эвристики + небольшая classifier-модель) на недоверенных входах, которые увидит LLM.
- Используйте строгие JSON-схемы, чтобы маркировать границу между «инструкциями» и «данными» — парсер отказывается от неоднозначных структур.
- Совпадения prompt-injection логируйте как security-событие, а не как шум.

## Sandbox и изоляция

- MCP-серверы запускаются с наименьшими привилегиями — отдельный пользователь, отдельный ФС-namespace, ограниченные capabilities.
- Любой сервер, касающийся хоста (файлы, команды, сеть), добавляет sandbox-слой: `gVisor`, Kata Containers, Firecracker, SELinux / AppArmor-профили, seccomp-фильтры.
- LLM-сгенерированный код — считаем враждебным. Никогда не исполняем его с полными правами пользователя; минимальный жизнеспособный sandbox — контейнер на вызов без доступа к хосту.

## Криптографическая верификация

- Подписывайте код сервера и публикуйте SBOM.
- Клиент проверяет подпись до загрузки сервера. Неизвестный издатель → отказ по умолчанию.
- TLS на всём, что по сети.
- Remote attestation там, где этого требует модель угроз — сервер доказывает, что запускает ожидаемый код.

## Транспорт

### `stdio` (локально)

- Предпочтительный для локальных MCP-серверов — убирает сетевые классы атак (DNS-rebinding, cross-origin-утечки).
- Pipe-коммуникация — всё ещё IPC-канал; сэндбоксьте сервер, чтобы он не ушёл из своего процесса.

### HTTP streaming (удалённый)

Следующие контроли обязательны, иначе не гоняйте MCP через HTTP:

- Лимиты payload — на запрос, на сообщение, на вложенность, на суммарный streaming-window. Защита от large-payload- и recursive-payload-DoS.
- Rate-limit — на tool, на клиента, на identity.
- Взаимная аутентификация — и клиент, и сервер предъявляют креды.
- mTLS — требуется клиентский сертификат, проверяется серверный.
- TLS-шифрование (предпочтительно 1.3).
- CORS — явный allow-list origin; никогда `*`.
- CSRF-защита — требование custom-заголовка, SameSite-cookie для сессионной модели.
- Целостность — счётчики сообщений, nonce или подписанные ответы, блокирующие replay и подделку.

## Безопасный дизайн tool-ов и UX

- Каждый tool — single-purpose с явным контрактом. Избегайте «сделай что угодно» tool-ов, чей scope зависит от промпта — они становятся усилителями prompt-injection.
- LLM не принимает authz-решения сам. Сервер enforce-ит authz независимо.
- **Two-stage commit** для high-impact операций: draft/preview-вызов возвращает ticket-ID и показывает точное действие; второй commit-вызов с ticket-ом его выполняет.
- Предусмотрите пути отката (draft-ID, снапшоты, обратимые операции). Ставьте время жизни commit — ticket истекает.

## Human in the loop

- Требуйте подтверждение рискованных операций. Используйте MCP-фичу «elicitation», чтобы сервер запрашивал approval через канал, который клиент обязан уважить.
- Подтверждающие сообщения чётко формулируют импакт: «Будут удалены 3412 записей из продовой БД».
- Не полагайтесь **только** на подтверждение человеком. Уставший пользователь прокликивает. Сочетайте с ограниченной властью — tool не может сделать ничего такого, что сервер отказался бы выполнить даже с одобрением.

## Логирование и наблюдаемость

- Логируйте: какой tool вызван, параметры (с редактированием секретов), исходный промпт, использованную идентичность, результат.
- OpenTelemetry-трейсы, связывающие вызов LLM → вызов tool → downstream-действие. End-to-end связность необходима для расследований.
- Неизменяемое, контрол-доступом защищённое хранилище логов. Идентичность, пишущая логи, не может их изменить.

## Паттерны развёртывания

### Полностью локально (stdio или http на localhost)

- Безопасность зависит от гигиены хоста.
- Предпочитайте `stdio`, чтобы избежать DNS-rebinding.
- Sandbox ограничивает blast-радиус.
- Ок для личного использования и разработки; не для multi-user-прода.

### Single-tenant remote (http)

- Аутентификация клиент-сервер **обязательна**.
- Безопасное хранение кредов — OS keychain, secret-менеджер. Не `~/.config/*.json`.
- Шифрованная и аутентифицированная связь end-to-end.
- Enterprise-клиенты принудительно резолвят discovery серверов по явному allow-list.

### Multi-tenant remote (http)

- Tenant-изоляция — жёсткое требование: identity на тенанта, ключи шифрования на тенанта, строгий RBAC внутри сервера.
- Предпочитайте серверы, хостящиеся самим провайдером сервиса (ему уже доверяете данные).
- Remote attestation, где возможно — клиент проверяет, что сервер запускает ожидаемый код.

## Кратко

Безопасность MCP — это пересечение всех других правил (auth, authz, транспорт, парсинг, логирование) плюс два MCP-специфичных момента: prompt-injection через tool-schema и тот факт, что LLM — не слой авторизации. Проектируйте сервер так, чтобы ненадёжно ведущая себя модель всё равно была ограничена собственными контролями сервера.
