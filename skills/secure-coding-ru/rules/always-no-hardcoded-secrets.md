# Никаких захардкоженных секретов

## Правило

Учётные данные, лежащие в исходном коде, считаются скомпрометированными в тот же момент, как исходник закоммитили, скопировали, передали ИИ-агенту, напечатали в стектрейсе или положили в бэкап. Считайте, что репозиторий рано или поздно станет публичным — потому что на практике многие так и случаются: по ошибке или при смене политики.

Правило безусловное: **ничего из перечисленного не должно оказаться в файлах исходников, примерах конфигов, тестовых фикстурах, docstring-ах и комментариях.**

## Что считается секретом

Категории, которые в исходниках запрещены всегда:

- Любые пароли — сервисных учёток, пользователей БД, админок, тестовых аккаунтов, ведущих в реальную систему
- API-ключи, bearer-токены, personal access tokens, client secret от OAuth, подписывающие webhook-секреты
- Приватные ключи (RSA/EC/Ed25519), TLS-ключи серверов, приватные SSH-ключи, ключи подписи кода
- Connection string с встроенными кредами (`postgres://user:pw@host/db`, `mongodb://…`, `amqps://…`)
- Значения сессионных cookie, JWT, refresh-токены — даже просроченные
- Облачные access-ключи и session-токены (AWS, GCP, Azure, Cloudflare, …)
- Общие HMAC-ключи и ключи симметричного шифрования

Тестовая учётка, указывающая на *общее* тестовое окружение, — это по-прежнему настоящая учётка. Локальные креды для изолированного dev-контейнера или одноразовой эфемерной БД допустимы только если эта строка вне своей песочницы бесполезна.

## Формы, которые нужно узнавать в лицо

При чтении или генерации кода эти паттерны — красный флаг, на них стоит остановиться:

| Паттерн | Префикс примера |
|---------|-----------------|
| AWS access key | `AKIA…`, `ASIA…`, `AROA…`, `AIDA…`, `AGPA…` |
| GitHub token | `ghp_…`, `gho_…`, `ghs_…`, `ghu_…`, `ghr_…` |
| Stripe key | `sk_live_…`, `sk_test_…`, `pk_live_…`, `pk_test_…` |
| Google API key | `AIza…` (35 символов после префикса) |
| Slack token | `xoxb-…`, `xoxp-…`, `xoxa-…` |
| JWT | три base64url-сегмента через `.` (часто начинается с `eyJ…`) |
| PEM-ключ | блок, начинающийся с `-----BEGIN ... PRIVATE KEY-----` |
| Учётные данные в URL | `proto://username:password@host` |

Также настораживают:

- Любая переменная с именем `PASSWORD`, `SECRET`, `API_KEY`, `TOKEN`, `AUTH`, `PRIV_KEY`, `CREDS`, значение которой — строковой литерал
- Длинные непрозрачные base64- или hex-строки рядом с аутентификационным кодом
- Дефолтные значения вида `os.getenv("DB_PASS", "hunter2")` — фолбэк всё так же живёт в исходниках

## Что делать вместо этого

1. Читать значение из переменных окружения в рантайме, без in-code дефолта. Если переменной нет — громко падать на старте.
2. Брать из секрет-менеджера (AWS Secrets Manager, GCP Secret Manager, Azure Key Vault, HashiCorp Vault, Doppler, 1Password Connect, …). Ротировать через менеджер, а не через деплой.
3. На машинах разработчиков — локальный `.env`, добавленный в `.gitignore`, плюс закоммиченный `.env.example` с плейсхолдерами.
4. В CI — инжектить через секреты репозитория/организации; не выводить их в логи; маскировать в выводе.
5. Если секрет обязан присутствовать на этапе сборки (например, аутентификация в container registry) — использовать BuildKit secrets или аналог платформы, а не `ARG` / `ENV`, которые остаются в слоях.

## Если секрет всё-таки попал

Сначала ротация, потом удаление. Удаление файла в следующем коммите **не** убирает секрет из git-истории. Правильная последовательность:

1. Отозвать кред у издателя (консоль AWS, настройки GitHub и т. п.).
2. Сгенерировать замену и подать её по механизмам выше.
3. Если репозиторий общий и значение действительно должно исчезнуть — переписать историю (`git filter-repo` или BFG), затем принудительно пушить, предварительно согласовав с коллегами.
4. Просмотреть логи доступа по отозванному кредиту за окно экспозиции.

Ротация — не опция: считайте, что любой раскрытый кред был скрейпнут за минуты.

## При применении правила

Агент обязан в своём ответе перечислить все места, где секрет мог оказаться в коде, и подтвердить, что его вынесли наружу. Это не ритуал — один пропущенный случай обнуляет правило целиком.
