# DevOps, CI/CD и контейнеры

Пайплайн сборки и доставки — ценная цель: скомпрометируйте его раз — и каждый артефакт после этого с бэкдором. Это правило покрывает упрочнение пайплайна, упрочнение контейнеров, virtual patching как временную меру и флаги тулчейна для нативного C/C++.

## Упрочнение CI/CD-пайплайна

### Репозиторий

- Protected-branches для всего, что мёржится в прод. Required reviews, required status checks.
- Подписанные коммиты, в идеале проверяемые на merge.
- Доступ к веткам — наименьшие привилегии; пушить в release-ветки могут только определённые сервис-аккаунты.

### Секреты внутри пайплайна

- Никогда не хардкодить в workflow-файлах. Тянуть из секрет-стора в рантайме.
- Маскировать в логах. В большинстве CI есть встроенное маскирование — пользуйтесь.
- Токены — строго по скоупу шага. CI-job, которому нужно только опубликовать артефакт, не нуждается в read-токене для issue-трекера.
- Предпочитайте короткоживущие OIDC-токены (GitHub OIDC → AWS, GitLab → GCP и т. п.) долгоживущим статическим секретам.

### Runner-ы

- Эфемерные, изолированные — каждому job-у свой контейнер/ВМ, по завершении — в утиль.
- Минимум прав на identity раннера. Сверхпривилегированный раннер — сверхпривилегированный атакующий при компрометации.
- Никаких общих кэшей, пересекающих границы доверия (запуск merged PR не должен видеть секреты из тестов ещё не смёрженной ветки).

### Security-гейты

В пайплайне, до merge/деплоя:

- **SAST** — статический анализ безопасности исходников
- **SCA** — software composition analysis по дереву зависимостей
- **IaC-сканирование** — policy-чеки на Terraform, CloudFormation, Helm, Kustomize
- **Secrets scanning** — pre-commit и on-PR
- **DAST** — динамическое тестирование против staging
- **Container scanning** — при сборке образа и при admission

Настройте пороги критичности: критические находки блокируют merge. Override требует документированного обоснования и компенсирующего контроля.

### Зависимости

- Lockfile в VCS и проверяется в пайплайне.
- Проверка целостности (`npm ci` уважает `package-lock.json`; `pip` умеет `--require-hashes`).
- Приватные registry для внутренних пакетов; избегайте wildcard-публичных.
- Пинните базовые образы и прочие transitive-зависимости по digest, где инструменты позволяют.

### Подпись и provenance

- Подписывайте коммиты, контейнерные образы, релизные артефакты.
- Проверяйте подписи на стороне деплоя — не «мы подписываем», а «мы отказываемся деплоить неподписанное».
- Внедряйте SLSA-style provenance: система сборки выпускает аттестацию о том, что и из какого источника, каким тулчейном собрано.

## Упрочнение Docker и контейнеров

### Пользователь и привилегии

- Не запускайтесь от root. Создайте непривилегированного пользователя в Dockerfile и `USER` на него.
- Никогда не с `--privileged`.
- Сбросьте все Linux-capabilities и верните только нужные: `--cap-drop=ALL --cap-add=NET_BIND_SERVICE` (в идеале слушайте на порту выше 1024, чтобы и эта не требовалась).
- `--security-opt=no-new-privileges`, чтобы заблокировать setuid-эскалацию.

### Daemon-socket

Никогда не монтируйте `/var/run/docker.sock` в контейнер. Это полная эскалация на хост. То же касается host-path-mount-ов cgroup-иерархии, Kubernetes-сокетов и всего, что выпускает процесс из контейнера.

Не выставляйте Docker daemon через TCP без mTLS. `-H tcp://0.0.0.0:...` без TLS — однокликовый takeover.

### ФС и сеть

- Root-ФС только на чтение (`--read-only`); временное write-пространство — через `tmpfs`-маунты, где нужно.
- Лимиты CPU и памяти на каждом контейнере — неограниченный контейнер сожрёт хост.
- Избегайте host-networking. Используйте user-defined bridge; выставляйте только реально нужные порты.

### Образы

- Минимальная база: `distroless`, `alpine` или scratch там, где хватает.
- Пин по digest. Тег (`node:20`) даёт later-push возможность поменять то, что вы запускаете.
- Multi-stage build-ы: компилируйте в большом образе, копируйте артефакт в минимальный рантайм-образ. Компиляторы, шеллы и пакет-менеджеры в финальный образ не попадают.
- `HEALTHCHECK` задан (либо эквивалент через оркестратор).
- `.dockerignore` исключает git-историю, IDE-метаданные, кэши, env-файлы.

### Секреты в образах

Не зашивайте секреты в слои — они навсегда в истории, даже если последующий слой их удалил. Используйте BuildKit secrets (`--mount=type=secret`) на билде и механизм секретов оркестратора в рантайме.

### Сканирование

- Сканируйте каждый образ на сборке (в CI).
- Сканируйте снова на admission (на стороне кластера), потому что безопасное вчера может иметь CVE сегодня.
- Высокие находки блокируют деплой, если нет явного waiver с тикетом.

## Node.js в контейнерах

Шаблон:

```dockerfile
FROM node:20-alpine@sha256:...   # пин по digest
WORKDIR /app

# Сначала зависимости — лучший кэш
COPY --chown=node:node package.json package-lock.json ./
RUN npm ci --omit=dev

COPY --chown=node:node . .

ENV NODE_ENV=production
USER node

# Корректный init и пересылка сигналов
ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "server.js"]
```

- `npm ci`, не `npm install` — детерминировано.
- `--omit=dev` — dev-зависимости не попадают в рантайм-образ.
- `NODE_ENV=production` — многие библиотеки смотрят на эту переменную.
- Пользователь `node`, не root.
- Init-процесс для аккуратного shutdown и пересылки сигналов.

## Virtual patching

Virtual patch — правило WAF / IPS, блокирующее эксплуатацию известной уязвимости на время, пока реальный фикс готовится. Временная мера, не замена фиксу.

- Держите WAF-развёртывание готовым до того, как оно понадобится — не в инциденте спешить.
- Предпочитайте positive (allow-list) правила для точности.
- Разворачивайте сперва в **log-only**; наблюдайте; затем enforce.
- Каждому virtual-patch-у — ссылка на CVE или тикет. Снимайте патч, когда код починен и задеплоен — просроченные virtual-патчи дрейфуют и плодят ложные срабатывания.

## Упрочнение тулчейна C / C++

При сборке нативного кода включайте компиляторные митигации — они почти бесплатны и покрывают классы багов, которые легко сделать:

### Флаги компилятора

- `-Wall -Wextra -Wconversion` — предупреждения о типичных ошибках.
- `-fstack-protector-all` или минимум `-fstack-protector-strong` — stack-canaries.
- `-fPIE -pie` — Position-Independent Executables для ASLR.
- `-D_FORTIFY_SOURCE=2` (или `=3` на свежей glibc) — compile-time + runtime bounds-check на набор библиотечных функций.
- `-fsanitize=cfi` с LTO — Control Flow Integrity, ловит indirect-call hijack.

### Флаги линкера

- `-Wl,-z,relro -Wl,-z,now` — RELRO/now, чтобы GOT стал read-only после старта.
- `-Wl,-z,noexecstack` — non-executable stack.
- NX / DEP и ASLR включены на уровне ОС.

### Debug vs release

- Sanitizers (`-fsanitize=address,undefined`) — в debug, off в release.
- Hardening-флаги — в release; assertions — только в debug.
- CI-шаг с `checksec`, падающий, если релизный бинарник без ожидаемых защит.

## Чек-лист реализации

- Секреты тянутся из vault-а в рантайме; никогда не хардкод в пайплайне.
- Runner-ы эфемерные; токены узкие; предпочитается короткоживущий OIDC.
- Security-сканы (SAST, SCA, DAST, IaC, container, secrets) гейтят merge; критика блокирует.
- Артефакты и коммиты подписаны; подписи проверяются до деплоя; provenance фиксируется.
- Контейнеры запускаются не от root, read-only FS, capabilities сброшены, лимиты есть. Никаких daemon-socket-маунтов.
- Образы минимальны, запинены по digest, сканируются на сборке и admission, с healthcheck-ом.
- Node.js-образы используют `npm ci`, production env, non-root, корректный init.
- Процесс virtual-patching-а определён, правила снимаются после code-fix.
- Для нативных сборок включены документированные hardening-флаги и проверяются в CI.
