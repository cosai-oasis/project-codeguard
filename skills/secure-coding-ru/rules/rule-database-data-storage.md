# Базы данных и хранилища данных

Покрывает конфигурационную поверхность SQL- и NoSQL-хранилищ — размещение в сети, транспорт, аутентификация, креды, права, упрочнение, платформенные нюансы. Защиту на уровне запросов (параметризация, SOQL, operator-injection в NoSQL) см. в `rule-input-validation-injection.md`.

## Размещение в сети

- Не выставляйте БД в Интернет. Привязывайте к приватному VPC / VNET / приватной подсети.
- Приложение и БД — по возможности на разных сетевых сегментах. DMZ для app-слоя и БД-слой, принимающий соединения только от app-слоя, — базовый паттерн.
- Отключайте TCP-доступ, когда приложение живёт на том же хосте и работает локальный сокет / named pipe.
- Если нужен TCP — слушайте на конкретном интерфейсе (не `0.0.0.0`) и ограничивайте source-адреса правилами firewall / security-group.
- Не пускайте толстые клиенты (desktop-приложения, BI-инструменты) напрямую в прод. Ставьте спереди аутентифицированный сервис.

## Безопасность транспорта

- Принимайте только TLS-соединения. Plaintext — отключить.
- TLS 1.2 и выше, с современными cipher suites (см. `rule-additional-cryptography.md`).
- Клиенты валидируют серверный сертификат — не `sslmode=require` (он часто значит лишь «использовать TLS, если доступен»), а `verify-full` или эквивалент, проверяющий хост и CA.
- Сертификаты на сервере БД подписаны внутренним CA или публичным — по обстоятельствам; не self-signed с клиентским override-ом.

Шифруется весь трафик, включая initial authentication handshake, а не только данные после auth.

## Аутентификация

- Аутентификация требуется всегда, даже для локальных / loopback-соединений. Неправильно настроенный локальный сервис — вектор privilege-escalation.
- Каждое приложение / сервис получает свой DB-пользователь. Не шарьте креды между сервисами.
- Креды сильные, уникальные, ротируются — особенно при оффбординге сотрудников или подозрении на компрометацию.
- Учётка удаляется, когда приложение, которому она принадлежала, уходит. Орфан-сервис-аккаунт — бэкдор.

## Хранение кредов

Креды — в secret-менеджере (см. `always-no-hardcoded-secrets.md`), не в исходниках.

Если файловый конфиг неизбежен:

- Кладите вне document-root любого веб-сервера.
- Ставьте права ФС так, чтобы файл читал только процесс DB-клиента.
- Никогда не коммитьте. `.gitignore` на путь *и* pre-commit-хук, сканирующий known-secret-shape.
- Шифруйте, если платформа позволяет (Linux keyring, macOS Keychain, Windows DPAPI).

Предпочитайте в порядке убывания: **managed identity / workload identity (IMDS, GCP service account, Azure MSI) → secret-менеджер → шифрованный конфиг → переменные окружения → файл → литерал в коде (никогда).**

## Модель прав

- Наименьшие привилегии на уровне учётки. У app-аккаунта есть только те права, что реально используются. `SELECT`, `INSERT`, `UPDATE`, `DELETE` на конкретные таблицы — не `ALL` на `*`.
- Не используйте встроенные административные учётки (`root`, `sa`, `SYS`, `postgres`) для приложения. Они нужны для администрирования; их blast-radius тотален.
- Не давайте app-аккаунту админских прав или ownership-а БД. Object ownership можно использовать для обхода row-level security и эскалации.
- Ограничивайте хосты, с которых может подключаться аккаунт (`CREATE USER ... @'10.0.0.0/24'` в MySQL, `pg_hba.conf` в PostgreSQL).
- Разделяйте окружения — dev, staging, production — отдельными учётками, отдельными БД и жёсткой сетевой границей. Staging-аккаунт с продовыми кредами — не разделение.
- Используйте table-, column- и row-level permissions там, где схема позволяет (`GRANT SELECT (public_columns) ON ...`, `CREATE POLICY ...` в PostgreSQL, VPD в Oracle).

## Упрочнение конфигурации

- Держите БД пропатченной. Критические CVE к DB-движкам эксплуатируются быстро.
- Запускайте сервис БД под непривилегированной OS-учёткой, не root.
- Удалите дефолтные аккаунты, sample-базы и дефолтные пароли, которые шлёт движок из коробки.
- Transaction-логи — на отдельном диске от data-файлов, с корректными правами.
- Регулярные шифрованные бэкапы с контролем доступа и периодическими учениями по восстановлению.
- Отключайте хранимки и фичи движка, которые вы не используете — `xp_cmdshell`, CLR-execution, external-tables, смотрящие в сеть, и т. п.
- Там, где требует комплаенс или threat-model — включайте database activity monitoring. Алёрт на неудачные логины, смены прав, крупные экспорты.

## Платформенные заметки

### SQL Server

- Отключайте `xp_cmdshell`, если он не нужен (почти никогда не нужен).
- Отключайте CLR-execution, если конкретная фича не требует.
- Выключайте SQL Browser service.
- Для людей — Windows Authentication или Azure AD; SQL-логины — только для сервисов.
- Mixed Mode Authentication off, если не обосновано.

### MySQL / MariaDB

- Прогоняйте `mysql_secure_installation` на каждой новой инстанции.
- У app-пользователей отключайте привилегию `FILE` — она позволяет читать/писать произвольные файлы от имени MySQL.
- Отключайте `LOCAL INFILE` в клиенте, если не нужен.
- Проверяйте `skip-networking` / `bind-address` на dev-инстансах.

### PostgreSQL

- `pg_hba.conf` ограничивает host-based auth; по умолчанию — `scram-sha-256` для пароля, `cert` для клиентского сертификата.
- `listen_addresses` — только приватный интерфейс.
- Row-level security политики на tenant-scoped таблицах.
- `log_statement` тюньте так, чтобы не логировать значения параметров (запросы с кредами в параметрах).

### MongoDB

- Аутентификация включена (`--auth`). Легаси-дефолт «open to localhost» наделал много взломов.
- `bindIp` — только приватный интерфейс.
- SCRAM-SHA-256; LDAP или x.509 — для enterprise.
- Field-level encryption там, где требует модель угроз.
- Отключайте неиспользуемые интерфейсы (HTTP-status, REST).

### Redis

- Аутентификация включена (`requirepass` или ACL).
- Переименовывайте / отключайте опасные команды (`CONFIG`, `FLUSHALL`, `DEBUG`, `SCRIPT`) на продовых инстансах.
- TLS терминируется на стороне Redis для любого cross-host трафика.
- Не выставляйте `6379` в публичный Интернет — исторически один из самых сканируемых портов.

## Кратко

Положите БД в приватную сеть, заставьте TLS, аутентифицируйте каждое соединение, храните креды в менеджере, выдавайте минимум прав, упрочняйте дефолтную конфигурацию движка и мониторьте изменения. Каждый слой блокирует свой класс атакующих — ни один сам по себе не достаточен.
